Segment 1 — Liveness

Liveness Probes

❤️ What It Does

  • Checks if container is alive
  • Failure → kubelet restarts the container
  • Does NOT remove from Service endpoints
  • Use for: deadlock, stuck threads

⚙️ Probe Types

  • httpGet — GET /healthz expects 2xx
  • exec — run command, exit 0 = healthy
  • tcpSocket — port must accept connection
  • grpc — gRPC health protocol
livenessProbe: httpGet: path: /healthz port: 8080 initialDelaySeconds: 30 # wait before first check periodSeconds: 10 # check every 10s failureThreshold: 3 # restart after 3 failures
Segment 2 — Readiness

Readiness Probes

✅ What It Does

  • Checks if container is ready to serve traffic
  • Failure → removed from Service endpoints
  • Does NOT restart the container
  • Use for: DB connection, cache warm-up

🔄 Traffic Flow

Service
→
Ready Pod ✅
Service
✗
Not Ready ❌
🎯 Zero-downtime deploys: New pods only receive traffic once readiness passes. Old pods stay in rotation until new ones are ready. This is how rolling updates work safely.
Segment 3 — Startup

Startup Probes

🚀 Why They Exist

  • Some apps take 2-3 min to start (JVM, large DBs)
  • Liveness probe would kill them before they're ready
  • Startup probe disables liveness/readiness until success
  • Success = app fully started, hand over to liveness

⚙️ Configuration

startupProbe: httpGet: path: /startup port: 8080 failureThreshold: 30 periodSeconds: 10 # Allows 300s (30×10) to start
💡 Rule of thumb: Set failureThreshold × periodSeconds to your worst-case startup time. Once startup succeeds, liveness & readiness take over normally.
Segment 4 — PDBs

PodDisruptionBudgets (PDB)

🛡️ What is a PDB?

  • Limits simultaneous voluntary disruptions
  • Protects during node drains, upgrades, rollouts
  • minAvailable — at least N pods must be up
  • maxUnavailable — at most N pods can be down
  • Enforced by drain/eviction API

📄 PDB Example

apiVersion: policy/v1 kind: PodDisruptionBudget metadata: name: api-pdb spec: minAvailable: 2 # or "80%" selector: matchLabels: app: api
🎯 SRE Rule: Every production Deployment with replicas > 1 should have a PDB. Without it, a node drain could take down all replicas simultaneously.
Knowledge Check

Quick Quiz 🎯

1. A pod's liveness probe fails 3 times. What happens?
kubelet restarts the container. The pod stays on the same node, same IP, but the container process is killed and restarted.
2. What is the difference between liveness and readiness probe failure?
Liveness failure → container restarted. Readiness failure → pod removed from Service endpoints (no traffic), container keeps running.
3. You have 5 replicas and a PDB with minAvailable: 4. A node drain tries to evict 3 pods. What happens?
Drain evicts 1 pod (5→4, still meets minAvailable=4), then blocks. The remaining 2 evictions would violate the PDB, so drain waits.
4. Why use a startup probe instead of just increasing initialDelaySeconds on liveness?
initialDelaySeconds adds fixed delay for ALL starts, even fast ones. Startup probe adapts — liveness only activates after startup succeeds, regardless of how long it took.
