Segment 1 — Why Distributed Tracing?

The Microservices Problem

Monolith vs Microservices

  • › Monolith: Single request → code path is linear, easy to debug
  • › Microservices: Single request → 10+ services in parallel/sequence
  • › Problem: Where is latency? API slow? DB? Network? Service B?
  • › Metrics alone: Tell you something is slow, not WHERE
  • › Logs alone: Scattered across services, hard to correlate

Tracing Solves This

  • › End-to-end visibility: Every service hop in one view
  • › Latency breakdown: See time spent in each service
  • › Failure root cause: Which service failed? Where?
  • › Performance optimization: Find bottlenecks fast
  • › Dependency mapping: Understand service relationships
Segment 2 — Core Concepts

Tracing Concepts

Trace Anatomy

# Single HTTP request creates ONE TRACE with multiple SPANS
TraceID: abc123xyz789
├─ Span (API Gateway) [0ms-50ms]
│ └─ SpanContext: TraceID=abc123xyz789, SpanID=span1
├─ Span (Auth Service) [5ms-15ms] parent=span1
│ └─ Attributes: user_id=42, auth_method=oauth
├─ Span (User DB) [10ms-12ms] parent=span1
│ └─ Events: query_start, query_end
└─ Span (Cache) [20ms-22ms] parent=span1
   └─ Error: connection refused

Trace

  • • Single request = 1 trace
  • • Has unique TraceID
  • • Contains N spans
  • • Flows across services

Span

  • • Unit of work in a service
  • • Has unique SpanID
  • • Parent-child relationships
  • • Has start time + duration

SpanContext

  • • Carries TraceID + SpanID
  • • Propagates across services
  • • In HTTP headers (W3C)
  • • Includes baggage (metadata)
Segment 3 — Jaeger Architecture

Jaeger Components

Data Flow

Application —[emit spans]→ Jaeger Agent (localhost:6831) —[UDP]→ Collector
↓
Storage (Elasticsearch / Cassandra / Badger) ←——————→ Query UI (http://localhost:16686)

Jaeger Agent

  • • Runs as sidecar/DaemonSet
  • • Listens on UDP 6831
  • • Buffers spans locally
  • • Ships to collector batch

Collector

  • • Central ingestion point
  • • Receives spans via gRPC/HTTP
  • • Validates + enriches spans
  • • Persists to storage backend

Query UI

  • • Web frontend
  • • Search traces by TraceID
  • • Visualize span timing
  • • Analyze dependencies
Segment 4 — OpenTelemetry

Instrumenting with OpenTelemetry

OTel Basics

  • › Vendor-neutral: Not locked to Jaeger
  • › SDK: Language-specific (Python, Go, Java, JS, etc.)
  • › Auto-instrumentation: Attach to frameworks automatically
  • › Manual spans: Add custom business logic spans
  • › Exporters: Send to Jaeger, DataDog, AWS X-Ray, etc.

Manual Instrumentation

# Python example
from opentelemetry import trace

tracer = trace.get_tracer(__name__)

with tracer.start_as_current_span("my_span") as span:
  span.set_attribute("user_id", 42)
  span.add_event("db_query_started")
  # ... do work ...
  span.set_status(Status(StatusCode.OK))
Segment 5 — Sampling Strategies

Sampling: Cost vs Visibility

Head-Based Sampling

  • › Probabilistic: Sample X% of all traces
  • › Decide at trace start (fast)
  • › May miss slow traces (sampled out)
  • › Good for: high-volume services
  • › Example: 10% sampling = 10% of traces

Tail-Based Sampling

  • › Wait for trace completion: Then decide to sample
  • › Sample slow/error traces at 100%
  • › Sample normal traces at 1%
  • › Good for: finding interesting traces
  • › Requires OTel Collector (more complexity)
Quiz

Jaeger & Tracing Quiz ⚡

Q1. What is a Trace?
✅ A single request flowing through your system. It has a unique TraceID and contains multiple spans from different services.
Q2. What is the difference between a Trace and a Span?
✅ A Trace is the entire request journey. A Span is a single unit of work within one service. One Trace = multiple Spans.
Q3. How does a service know which trace to belong to?
✅ Via SpanContext propagation — the calling service passes TraceID in HTTP headers (W3C traceparent standard), so downstream services know they're part of the same trace.
Q4. What is OpenTelemetry?
✅ A vendor-neutral, open-source standard and SDK for distributed tracing and metrics instrumentation. Lets you instrument once, export to any backend (Jaeger, DataDog, etc.)
Q5. What's the downside of head-based sampling?
✅ You might sample out slow/error traces by random chance. If you sample at 1%, you only capture 1% of errors — you might miss the one that matters.
