Segment 1 — HPA

HPA: Horizontal Pod Autoscaler

Scale replicas based on metrics

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

Important

  • › HPA requires resource requests to calculate utilization
  • › Metrics: CPU, memory, custom (Prometheus), external (SQS depth)
  • › Metrics Server must be installed in cluster
Segment 2 — VPA

VPA: Vertical Pod Autoscaler

Vertical Scaling

  • › Adjusts CPU/memory requests + limits
  • › Uses histograms of actual usage
  • › Modes: Off, Initial, Auto, Recreate
  • › Auto mode evicts pods to apply new sizes
  • › Good for right-sizing to actual usage

VPA Example

apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: myapp-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  updateMode: Auto
Segment 3 — KEDA

KEDA: Event-Driven Autoscaling

Scale to zero based on events

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: myapp-scaler
spec:
  scaleTargetRef:
    name: myapp
    kind: Deployment
  minReplicaCount: 0
  maxReplicaCount: 20
  triggers:
  - type: sqs
    metadata:
      queueURL: https://sqs.us-east-1.amazonaws.com/123/myqueue
      queueLength: 5

Supported Triggers

  • › SQS queue depth, Kafka lag, Prometheus, cron, webhooks
  • › Scale replicas from 0 when queue has messages
  • › Perfect for event-driven workloads, batch jobs
Segment 4 — Cluster Autoscaler

Cluster Autoscaler

Node Provisioning

  • › Adds nodes when pods are unschedulable
  • › Removes nodes when underutilized
  • › Works with node groups (ASG in AWS)
  • › Slow: takes minutes to provision nodes

CA Behavior

  • › Scale-up — when nodes unavailable
  • › Scale-down — after 10 min underutilized
  • › Skip nodes with system pods, local storage
  • › Respects PDBs (Pod Disruption Budgets)
Segment 5 — Karpenter

Karpenter: Next-Gen Node Provisioning

Karpenter NodePool

apiVersion: karpenter.sh/v1beta1
kind: NodePool
metadata:
  name: default
spec:
  template:
    spec:
      requirements:
      - key: node.kubernetes.io/capacity-type
        operator: In
        values: [spot, on-demand]

Karpenter Advantages

  • › Faster scaling (seconds vs minutes)
  • › Bin-packing — efficient pod placement
  • › Diverse instance types automatically
  • › Spot + On-Demand fallback

vs Cluster Autoscaler

  • › CA: Generic, slow, uses ASGs
  • › Karpenter: Intelligent, fast, AWS-native
  • › Karpenter: Better cost optimization
Segment 6 — Best Practices

Autoscaling Best Practices

Do's

  • › Set resource requests — HPA and Karpenter need them
  • › Use HPA + node autoscaler — both needed for full scaling
  • › KEDA for event-driven — queues, webhooks
  • › Karpenter for new clusters — superior to CA

Don'ts

  • › HPA + VPA on same resource — conflicts
  • › No requests/limits — HPA can't scale
  • › Scale to zero without handling graceful shutdown
Quiz

Autoscaling Quiz ⚡

Q1. HPA needs what to calculate CPU utilization?
✅ Resource requests. HPA uses actual CPU divided by requested CPU to get percentage.
Q2. VPA evicts pods to resize them. What mode does this?
✅ Auto mode. Off mode only recommends; Initial only sets on pod creation; Recreate also evicts.
Q3. KEDA ScaledObject with minReplicaCount: 0. When are pods created?
✅ When the trigger fires (e.g., messages in SQS queue). Otherwise 0 pods run to save costs.
Q4. Cluster Autoscaler vs Karpenter: which is faster?
✅ Karpenter. It provisions nodes in seconds; CA takes minutes via ASG launch templates.
Q5. Can you use HPA and VPA on the same Deployment?
✅ No. HPA scales replicas (horizontal), VPA scales pod size (vertical). They conflict and should not run together.
