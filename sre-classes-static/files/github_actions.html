GitHub Actions Concepts

Event: Trigger (push, PR open, schedule)
Workflow: YAML file in .github/workflows/
Job: Set of steps running on a runner
Step: Single task (run command or use action)
Runner: Machine executing the workflow (GitHub-hosted or self-hosted)
Action: Reusable unit of code (actions/checkout@v4)

Workflow Syntax

name: CI on: push: branches: [main, develop] pull_request: branches: [main] schedule: - cron: '0 2 * * *' # Daily at 2am UTC workflow_dispatch: # Manual trigger jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: npm install - run: npm test

Jobs & Steps

jobs: build: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: 18 - run: npm run build deploy: runs-on: ubuntu-latest needs: build # Wait for build job steps: - run: echo "Deploying..."

Sequential execution: Use needs to create dependencies

Triggers & Events

push (with filters)

Trigger on commit to branches/paths: branches: [main], paths: ['src/**']

pull_request

Trigger on PR open/update: types: [opened, synchronize]

schedule (Cron)

Periodic runs: cron: '0 2 * * *' (2am UTC daily)

workflow_dispatch

Manual trigger with optional input parameters

release

Trigger on GitHub release creation

repository_dispatch

External API trigger via webhook

Secrets & Environment Variables

jobs: deploy: runs-on: ubuntu-latest environment: production # Protection rule env: LOG_LEVEL: debug steps: - run: echo $LOG_LEVEL - run: curl -H "Authorization: token ${{ secrets.GITHUB_TOKEN }}" - run: echo "VERSION=1.0" >> $GITHUB_ENV - run: echo "${{ env.VERSION }}"

Environment protection rules: Require approvals for prod deployments

Matrix Builds

jobs: test: runs-on: ${{ matrix.os }} strategy: matrix: node-version: [16, 18, 20] os: [ubuntu-latest, windows-latest] fail-fast: false # Continue other versions steps: - uses: actions/setup-node@v4 with: node-version: ${{ matrix.node-version }} - run: npm test

Tests all combinations: 3 Node versions × 2 OSes = 6 parallel jobs

Docker Build & Push

- uses: docker/build-push-action@v5 with: context: . push: true registry: public.ecr.aws username: ${{ secrets.AWS_ACCESS_KEY_ID }} password: ${{ secrets.AWS_SECRET_ACCESS_KEY }} tags: ${{ env.REGISTRY }}/${{ github.repository }}:latest cache-from: type=gha cache-to: type=gha,mode=max

Also: Use reusable workflows with uses: ./path/workflow.yml

Knowledge Check

Q1: What is the purpose of the "needs" keyword in jobs?

It creates job dependencies — the specified job must complete successfully before the current job runs.

Q2: How do you trigger a workflow manually?

Use workflow_dispatch: in the on: section to enable manual trigger.

Q3: What is matrix in GitHub Actions?

Matrix allows testing across multiple versions/environments simultaneously, creating parallel job combinations (e.g., 3 Node versions × 2 OS = 6 jobs).

Q4: How are secrets accessed in workflow steps?

Via ${{ secrets.SECRET_NAME }} syntax. Secrets are masked in logs for safety.

Q5: What environment feature adds protection to production deployments?

Environment protection rules allow requiring approvals, restricting to specific branches, or using deployment gates before prod workflows run.
