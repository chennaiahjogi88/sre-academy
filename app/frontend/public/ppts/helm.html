What is Helm?

Helm is the package manager for Kubernetes — like apt, npm, or brew.

  • Versioning - Track versions of applications
  • Templating - Generate YAML from templates + values
  • Dependency Management - Define and manage chart dependencies
  • Rollbacks - Easily revert to previous versions
  • Repositories - Share and discover charts

Problem Helm Solves:

Instead of managing hundreds of YAML files, you manage reusable, configurable templates.

⎈ One chart = One application that can be deployed to many environments
Architecture

Helm Architecture

Three Core Concepts:

Chart

A Helm package

Templated YAML files + metadata

Release

Instance of a chart

Running deployment in K8s

Repository

Collection of charts

Public (Helm Hub) or private

Helm CLI → Chart Repo → values.yaml → templates/ → YAML → kubectl → K8s cluster

Helm

Chart Structure

A Helm chart is a directory with:

my-chart/ ├── Chart.yaml (chart metadata) ├── values.yaml (default values) ├── values-prod.yaml (production values) ├── templates/ │ ├── deployment.yaml │ ├── service.yaml │ ├── configmap.yaml │ ├── ingress.yaml │ ├── _helpers.tpl │ └── NOTES.txt ├── charts/ (dependencies) └── README.md
  • Chart.yaml - Version, name, description
  • values.yaml - Default configuration values
  • templates/ - YAML templates with {{ .Values.xxx }}
  • _helpers.tpl - Template functions and macros
Templates

Values & Templating

values.yaml (defaults):

replicaCount: 3 image: repository: nginx tag: 1.21.0 service: type: ClusterIP port: 80

Template file (templates/deployment.yaml):

apiVersion: apps/v1 kind: Deployment metadata: name: {{ .Release.Name }} spec: replicas: {{ .Values.replicaCount }} template: spec: containers: - image: {{ .Values.image.repository }}:{{ .Values.image.tag }}
Commands

Core Helm Commands

# Add a chart repository helm repo add bitnami https://charts.bitnami.com/bitnami # Install a chart (create release) helm install my-release bitnami/nginx # Upgrade a release helm upgrade my-release bitnami/nginx --values custom-values.yaml # Rollback to previous version helm rollback my-release 1 # List all releases helm list # Show release values helm get values my-release # Uninstall release helm uninstall my-release # Test chart (dry run) helm template my-release ./my-chart --debug
Best Practices

Helm Best Practices

  • --dry-run - Always test before applying: helm install --dry-run --debug
  • Pin Chart Versions - Don't use latest, lock to specific versions
  • Use Helmfile - Manage multiple charts with one file
  • Lint Charts - Check for errors: helm lint ./my-chart
  • Values Overrides - Use -f and --set for different environments
  • Semantic Versioning - Follow semver for chart versions
  • Document Values - Explain what each value does in values.yaml
  • Test Installs - Create test environments before production
Quiz

Knowledge Check

Q1: What are the three core concepts in Helm?
Chart (package), Release (running instance), Repository (chart collection).
Q2: What does the values.yaml file do?
Provides default configuration values that are injected into templates using {{ .Values.xxx }} syntax.
Q3: How do you test a Helm chart before installing?
Use helm install --dry-run --debug or helm template to see the generated YAML.
Q4: What command rolls back a release to a previous version?
helm rollback <release-name> <revision-number>
Q5: When should you use Helmfile?
When you need to manage multiple Helm charts and releases together as a single unit.
