Rolling Updates

Default Kubernetes strategy: gradually replace old pods with new ones

apiVersion: apps/v1 kind: Deployment spec: strategy: type: RollingUpdate rollingUpdate: maxSurge: 1 # 1 extra pod allowed maxUnavailable: 0 # 0 pods down at a time
Pros: No extra infrastructure, automatic K8s native
Cons: Mixed versions during rollout, slow if pod startup slow

Blue/Green Deployments

Two identical environments, instant traffic switch via LB/DNS

Blue Env (v1.0) → LB/DNS → Active ↓ (instant switch) Green Env (v2.0) → LB/DNS → Active (rollback to v1.0 instant)
Pros: Instant rollback, zero-downtime, no mixed versions
Cons: Double infrastructure cost, state synchronization needed
Use Case: Critical systems, large deployments, strict SLAs

Canary Deployments

Gradual traffic shift: 5% → 25% → 50% → 100%, monitor at each step

v1.0: ████████████████░░ (95%) v2.0: ██░░░░░░░░░░░░░░░░ (5%) ← Monitor error rate, latency Continue if healthy: v1.0: ███████░░░░░░░░░░░░ (75%) v2.0: █████████░░░░░░░░░░ (25%) Rollback if issues detected
Pros: Risk-aware, real-world traffic testing, automated rollback
Cons: Complexity, requires metrics/observability

ArgoCD Rollouts

apiVersion: argoproj.io/v1alpha1 kind: Rollout spec: strategy: canary: steps: - setWeight: 5 - pause: {duration: 5m} - setWeight: 25 - pause: {duration: 5m} - setWeight: 50 analysis: templates: - name: success-rate interval: 1m

Uses Prometheus metrics for automated promotion/rollback decisions

Feature Flags

Decouple Deploy from Release: Ship code but hide behind flags
Dark Launch: Test in production with 0% traffic before enablement
Gradual Rollout: Enable for 5% users, 25%, 50%, 100%
Kill Switch: Instantly disable buggy feature without redeployment
Tools: LaunchDarkly, Flagsmith, OpenFeature, custom services
// Application code if (featureFlags.isEnabled("new-checkout", user)) { // New checkout experience } else { // Old checkout experience }

Knowledge Check

Q1: What is the main benefit of canary deployments?

Gradual traffic shift with real-world monitoring enables risk-aware rollouts with automated rollback on detected issues.

Q2: When would you use blue/green over rolling updates?

For critical systems needing instant rollback, zero downtime, and no mixed versions during deployment.

Q3: What is the purpose of feature flags in deployments?

Decouple code deployment from feature release, enabling dark launches, gradual rollouts, and instant kill switches without redeployment.

Q4: Why must database migrations be backward-compatible?

To enable safe rollbacks — if old code is still running, it must understand the new schema or vice versa.

Q5: What ArgoCD Rollouts feature enables canary deployments?

The Rollout CRD with canary strategy, analysis templates (Prometheus metrics), and automated promotion/rollback decisions.
