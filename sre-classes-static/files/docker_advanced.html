Segment 1 — Storage

Volumes & Bind Mounts

Docker Volume
Managed by Docker
Stored in /var/lib/docker/volumes. Best for production — portable, backups, encryption
Bind Mount
Host Path
Maps a host directory into the container. Great for development — live code reload
tmpfs Mount
In-Memory
Stored in host memory only. Never persisted — good for sensitive temporary data
Volume Commands
Lifecycle
docker volume create/ls/inspect/rm — manage volumes independently of containers
Named Volume
-v db-data:/var/lib/mysql
Survives container removal. Share between containers. Best practice for databases
SRE Tip
Volumes for Stateful Apps
Never store DB data inside a container layer — use volumes or external managed services
Segment 2 — Networking

Docker Networking Modes

Network Drivers

  • › bridge (default) — isolated network, containers communicate by name
  • › host — container shares host network stack (no isolation, max performance)
  • › none — completely isolated, no networking
  • › overlay — multi-host networking (Swarm/K8s)
  • › macvlan — assign MAC address (legacy systems)

User-Defined Bridge

# Create custom network
docker network create mynet

# Containers find each other by NAME
docker run -d --name db \
--network mynet postgres

docker run -d --name app \
--network mynet \
-e DB_HOST=db myapp
Segment 3 — Best Practices

Production Best Practices

Security

  • › Never run as root — use USER instruction
  • › Use minimal base images (alpine, distroless)
  • › Scan images: docker scout / trivy
  • › Don't put secrets in Dockerfile or ENV
  • › Read-only filesystem where possible (--read-only)
  • › Drop Linux capabilities (--cap-drop ALL)

Performance & Size

  • › Multi-stage builds — only ship runtime
  • › Combine RUN commands with && to reduce layers
  • › Use .dockerignore — exclude node_modules, .git
  • › Pin base image versions (node:18.17.0-alpine)
  • › Set resource limits: --memory and --cpus
  • › Use HEALTHCHECK instruction
Segment 4 — Observability

Logging & Health Checks

Logging Drivers

  • › json-file (default) — local JSON files
  • › awslogs — ship to CloudWatch
  • › fluentd/fluentbit — log aggregation
  • › journald — systemd journal
  • › Always log to stdout/stderr in containers

HEALTHCHECK

HEALTHCHECK --interval=30s \
--timeout=5s \
--start-period=10s \
--retries=3 \
CMD curl -f \
http://localhost:3000/health \
|| exit 1

# Status: healthy / unhealthy / starting
Quiz

Volumes & Networking Quiz ⚡

Q1. What happens to data in a container layer when the container is removed?
✅ It is lost permanently. Always use Docker Volumes for persistent data that must survive container removal.
Q2. In a user-defined bridge network, how do containers communicate?
✅ By container name — Docker provides built-in DNS resolution. Container "app" can reach container "db" using the hostname "db".
Q3. Why avoid storing secrets in ENV variables in Dockerfile?
✅ ENV variables are baked into the image and visible via docker inspect or docker history. Use Docker Secrets or a secrets manager instead.
Q4. What is the "host" network driver?
✅ The container shares the host's network namespace — no port mapping needed, maximum performance, but no network isolation.
Q5. What should containers always write logs to?
✅ stdout and stderr — Docker captures these streams and routes them through the configured logging driver.
