Segment 1 — Why Compose?

The Multi-Container Problem

Without Compose

# Run each container manually...
docker network create app-net
docker volume create db-data
docker run -d --name postgres \
--network app-net \
-v db-data:/var/lib/postgres \
-e POSTGRES_PASSWORD=pass postgres
docker run -d --name redis \
--network app-net redis
docker run -d --name app \
--network app-net \
-p 8080:3000 -e DB_HOST=postgres myapp
# Repeat on every machine... 😩

With Compose

# One command 🎉
docker compose up -d

# Everything defined in compose.yaml:
- Services
- Networks (auto-created)
- Volumes
- Environment variables
- Dependencies (depends_on)
- Health checks
Segment 2 — Compose File

compose.yaml Anatomy

services:
  app:
    build: .                     # or image: myapp:1.0
    ports: ["8080:3000"]
    environment:
      DB_HOST: postgres
    depends_on:
      postgres: {condition: service_healthy}
  postgres:
    image: postgres:15-alpine
    environment:
      POSTGRES_PASSWORD: ${DB_PASS} # from .env file
    volumes: [db-data:/var/lib/postgresql/data]
    healthcheck:
      test: ["CMD","pg_isready"]
volumes:
  db-data: # named volume — survives compose down

Segment 4 — Overrides

Overrides & Profiles

Override Files

  • › compose.yaml — base config
  • › compose.override.yaml — auto-merged in dev
  • › compose.prod.yaml — production extras
  • › docker compose -f compose.yaml -f compose.prod.yaml up
  • › Dev: mount code for live reload, no replicas
  • › Prod: resource limits, secrets, replicas

Essential Commands

docker compose up -d     # start all
docker compose down      # stop & remove
docker compose down -v   # + remove volumes
docker compose logs -f app
docker compose exec app sh
docker compose ps
docker compose scale app=5
docker compose build     # rebuild images
Quiz

Docker Compose Quiz ⚡

Q1. What does depends_on with condition: service_healthy do?
✅ Waits until the dependency passes its HEALTHCHECK before starting the dependent service. (Basic depends_on only waits for container start, not readiness.)
Q2. How do services in the same Compose network find each other?
✅ By service name — Compose creates a network where each service name resolves to the container's IP. "app" can reach "postgres" using the hostname "postgres".
Q3. What is the difference between compose down and compose down -v?
✅ compose down stops and removes containers and networks. compose down -v also removes named volumes — this will delete your database data!
Q4. How do you pass secrets into Compose without hardcoding them?
✅ Use a .env file (not committed to git) with variable substitution: ${DB_PASS}. Docker Compose automatically loads .env from the project directory.
Q5. Can Docker Compose deploy to multiple servers?
✅ Not directly — Compose is single-host. For multi-host orchestration, use Kubernetes or Docker Swarm (with Compose file support via stack deploy).
