Burn Rate Concept

What is Burn Rate?

The rate at which you consume your error budget.

Burn Rate = 1: Consuming budget at exactly the SLO pace. At this rate, you'll exhaust budget at the end of the window.

Burn Rate = 2: Consuming budget at double speed. Budget will be gone in half the time.

Burn Rate = 10: Consuming budget 10x faster. If window is 30 days, budget exhausted in 3 days.

Burn Rate Calculation

Burn Rate = Error Rate / (1 - SLO)

Example 1: SLO = 99.9%, Error Rate = 1%

Burn Rate = 1% / (1 - 0.999) = 1% / 0.001 = 1000

You're consuming budget 1000x faster than the SLO pace. Immediate alert!

Example 2: SLO = 99.9%, Error Rate = 0.2%

Burn Rate = 0.2% / (1 - 0.999) = 0.2% / 0.001 = 200

Still burning fast — budget would be gone in a few hours.

Single Window Alerts: The Trade-Off

Long Window (1 hour)

  • Detects sustained problems ✓
  • Slower response ✗
  • Takes 1 hour to trigger

Short Window (5 minutes)

  • Fast response ✓
  • High false positive rate ✗
  • Noisy alerts tire teams

Problem: You can't have both low latency AND low false positives with a single window.

Multi-Window Alerting (Google's Approach)

Two-Window Strategy

  • Long Window (1 hour): Detects sustained burn
  • Short Window (5 minutes): Confirms current burn rate is high
  • Both must be true: Alert only if BOTH windows show high burn rate

Why This Works

  • 5-minute spike doesn't trigger alert (sustained)
  • High error for just 1 minute doesn't alert (confirms with 1h window)
  • Real incident burns on both timescales
  • Dramatically reduces false positives while keeping response time fast

Alert Severity Tiers

Severity Burn Rate Budget Exhaustion Action
Ticket 2.0x 3 days Create Jira ticket, page oncall next morning
Page 6.0x 1 day Page oncall immediately, investigate
Critical Page 14.4x 1 hour Emergency all-hands, war room immediately

Strategy: More severe burns get faster response. Proportional to the threat level.

PromQL for Multi-Window Burn Rate Alert

Two-Window Alert Rule (Prometheus)

groups: - name: sre-burn-rate-alerts rules: - alert: HighBurnRate expr: | ( # 1-hour window burn rate > 6 (page severity) sum(rate(requests_total{status=~"5.."}[1h])) / sum(rate(requests_total[1h])) / (1 - 0.999) >= 6 ) and ( # 5-minute window confirms high burn sum(rate(requests_total{status=~"5.."}[5m])) / sum(rate(requests_total[5m])) / (1 - 0.999) >= 6 ) for: 5m annotations: summary: "SLO burn rate is {{ $value }}x"

This alert requires BOTH windows to be true for 5 minutes. Reduces noise significantly.

Quiz: Burn Rate Alerts

Q1: What does burn rate = 1 mean?
A: You're consuming error budget at exactly the SLO pace. Budget will be exhausted at the end of the measurement window (30 days, etc.)
Q2: SLO = 99.9%, current error rate = 0.5%. What is the burn rate?
A: 0.5% / (1 - 0.999) = 0.5% / 0.001 = 500x. Consuming budget 500x faster than normal.
Q3: Why use a two-window alert instead of a single-window alert?
A: Two-window reduces false positives while keeping low latency. Requires sustained burn on both timescales (1h + 5m) before alerting.
Q4: What are the three alert severity tiers by burn rate?
A: Ticket (2x, 3d gone), Page (6x, 1d gone), Critical Page (14.4x, 1h gone)
Q5: Burn rate is 14.4x. How long until the monthly error budget is exhausted?
A: 30 days / 14.4 = approximately 2 hours. At this rate, monthly budget consumed in 2 hours.
