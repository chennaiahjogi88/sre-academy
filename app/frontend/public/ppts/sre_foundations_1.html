SRE vs DevOps

DevOps

  • Culture & Practice - bridging dev and ops
  • CI/CD Focus - rapid deployment pipelines
  • Automation - eliminate manual processes
  • Collaboration - shared responsibility
  • Infrastructure as Code - version everything

SRE

  • Reliability Engineering - systems thinking
  • SLOs & SLIs - quantified reliability
  • Error Budgets - balance innovation and stability
  • Toil Reduction - automate operational work
  • Blameless Culture - learn from failures
Principles

Google's SRE Book

Three Pillars of SRE:

  • Reliability - systems that stay up and perform
  • Scalability - handling growth without reinventing
  • Efficiency - resource optimization and cost control

The 4 Golden Signals:

  • Latency - response time of requests
  • Traffic - demand on the system
  • Errors - rate of failed requests
  • Saturation - how "full" the system is
Metrics

Reliability Fundamentals

Key Definitions:

  • MTTF (Mean Time To Failure) - average time before failure occurs
  • MTTR (Mean Time To Repair) - average time to fix after failure
  • MTBF (Mean Time Between Failures) - total time / number of failures

Availability Formula:

Availability (%) = MTBF / (MTBF + MTTR) × 100 99.9% = 8.7 hours downtime per year 99.99% = 52.6 minutes downtime per year 99.999% = 5.26 minutes downtime per year
Observability

The 4 Golden Signals

Latency

Response time of requests (p50, p95, p99)

Example: API response time < 100ms

Traffic

Request rate and demand on system

Example: Requests per second (RPS)

Errors

Rate of failed requests (4xx, 5xx)

Example: Error rate < 0.1%

Saturation

How "full" is the resource (CPU, memory, disk)

Example: CPU < 70%

Culture

Toil & Project Work

What is Toil?

  • Manual, repetitive, automatable work
  • No enduring value - must be repeated tomorrow
  • Scales linearly with system growth
  • Examples: Log rotation, manual deployments, password resets

Google's 50% Rule:

  • SRE teams should spend max 50% of time on toil
  • Remaining 50% on project work, improvements, automation
  • Reduces burnout and enables innovation

Quiz

Knowledge Check

Q1: What is the primary difference between SRE and DevOps?
DevOps is a culture/practice, while SRE is a discipline focused on reliability engineering with quantified objectives (SLOs, error budgets).
Q2: What is MTTR?
Mean Time To Repair - the average time it takes to fix a system after a failure occurs.
Q3: Name the 4 Golden Signals:
Latency, Traffic, Errors, and Saturation.
Q4: What is toil in SRE?
Manual, repetitive, automatable work with no enduring value that must be repeated.
Q5: What is Google's rule about toil allocation?
The 50% rule - SRE teams should spend max 50% of time on toil, rest on improvements.
