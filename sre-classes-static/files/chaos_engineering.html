Segment 1 — Principles

Chaos Engineering Principles

The Philosophy

  • › "Break things on purpose" to learn
  • › Netflix Chaos Monkey origin (2010s)
  • › Better to fail in labs than in production
  • › Hypothesis-driven experiments
  • › Document findings, improve systems

The 5 Principles

  1. Define steady state — metrics of normal system
  2. Form hypothesis — "System handles failure X"
  3. Introduce variable — inject the fault
  4. Prove/disprove — measure steady state
  5. Minimize blast radius — control the damage
Segment 2 — Steady State

Defining Steady State

Steady State = Baseline Metrics

Request Success Rate: 99.9% 4xx/5xx-free requests
Latency p99: < 200ms during normal traffic
Pod Availability: Replicas healthy, no evictions
Resource Usage: CPU/memory within range
Database Connections: Pool healthy, no timeouts

Why Steady State Matters

  • › Baseline to compare: "Did fault break system?"
  • › If steady state degrades during chaos, system not resilient
  • › Quantifies hypothesis success/failure
Segment 3 — Fault Types

Common Fault Types

Pod Faults

Pod kill, force restart, eviction simulation. Tests pod recovery and replica management.

Network Faults

Latency injection, packet loss, network partition. Simulate bad network conditions.

Resource Faults

CPU/memory stress, disk fill, I/O throttle. Stress container resources.

Dependency Faults

Database timeout, API 500s, DNS failure. External service failures.

Node Faults

Node drain, node NotReady. Cluster scaling resilience.

Time Faults

Clock skew, delay injection. Time-sensitive logic bugs.

LitmusChaos: CNCF Framework

ChaosEngine YAML

apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: pod-kill
spec:
  appinfo:
    appns: default
    applabel: app=myapp
  chaosServiceAccount: litmus-sa
  experiments:
  - name: pod-kill
    spec:
      components:
        env:
        - name: CHAOS_DURATION
          value: '30'

Segment 6 — GameDays

GameDays: Planned Chaos

GameDay Format

Planning (1-2 weeks before): Define scenarios, roles, success criteria
Staging Test (3 days before): Run scenarios in non-prod, validate assumptions
GameDay (2-4 hours): Cross-team exercise in staging or sandbox
Debrief (same day): Document findings, create action items
Follow-up (1 week): Implement fixes, measure improvement

GameDay Benefits

  • › Cross-team collaboration — eng, ops, support together
  • › Improve MTTD/MTTR — faster incident response
  • › Build confidence — team knows systems are resilient
  • › Runbook validation — test playbooks in safe environment
Quiz

Chaos Engineering Quiz ⚡

Q1. A chaos experiment kills all replicas of a service. What should happen?
✅ Deployment controller recreates them within seconds. If they don't reappear, system is not resilient.
Q2. Your steady state is "p99 latency < 100ms". During pod-kill chaos, p99 becomes 500ms. Pass or fail?
✅ Fail. Steady state degraded. System should survive pod kills with latency staying near baseline (or brief spike then recovery).
Q3. What does "minimize blast radius" mean in chaos?
✅ Limit the scope of failure. Start with 1 pod, not all. Run in staging, not production. Use tight time windows.
Q4. LitmusChaos uses which K8s resource to define an experiment?
✅ ChaosEngine. It references a ChaosExperiment and applies it to target pods via label selectors.
Q5. A GameDay runs a scenario: "Database goes unavailable for 5 min". What should teams do during this?
✅ Execute their incident response plan. Test alerting, runbooks, team coordination. Measure MTTD (time to detect). Debrief findings.
