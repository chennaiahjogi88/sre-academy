GitOps Principles

1. Declarative Infrastructure: Desired state in Git, versioned
2. Single Source of Truth: Git repo as authoritative source
3. Automated Reconciliation: System constantly syncs cluster to Git
4. Pull Model: Operator pulls from Git, not pushed externally
5. Auditable & Reversible: Full history in Git, easy rollbacks
6. Infrastructure as Code: Everything reproducible from Git

ArgoCD Architecture

Repo Server

Clones Git repo, renders Kustomize/Helm/Jsonnet manifests

Application Controller

Monitors cluster state, compares with Git, reconciles

API Server

Provides REST API, web UI, user management

Dex (SSO)

OAuth2/OIDC authentication integration

Redis Cache

Caches app states, repo data for performance

Notification Controller

Sends webhooks, Slack alerts on sync events

Sync Policies

Manual Sync

Requires explicit approval in UI/CLI to sync. Safe for prod, slower releases.

Automated Sync

Auto-syncs on Git commit. Fast, less control. Use with pull requests for reviews.

Prune

Deletes resources removed from Git. Prevents orphaned resources.

Self-Heal

Reverts manual cluster changes, re-syncs to Git state automatically.

App-of-Apps Pattern

# Root Application points to app directory apiVersion: argoproj.io/v1alpha1 kind: Application metadata: name: root-app spec: source: repoURL: https://github.com/myorg/apps path: apps # Directory of Application manifests destination: server: https://kubernetes.default.svc namespace: argocd # Each file in apps/ is an Application apps/ ├── app-auth.yaml ├── app-api.yaml ├── app-web.yaml

Use ApplicationSet for multi-cluster/multi-environment deployments

ArgoCD in Production

RBAC: Projects, AppProject CRD, user roles for access control
SSO/OIDC: Integrate with Okta, Azure AD, or any OIDC provider
Notifications: Webhooks to Slack, email on sync success/failure
Image Updater: Auto-update container images on new tags
High Availability: Run multiple replicas of API/Controller
Secret Management: Sealed Secrets, External Secrets Operator

Knowledge Check

Q1: What is the main principle of GitOps?

Git is the single source of truth for desired infrastructure state, and operators continuously reconcile the cluster to match.

Q2: What is the difference between manual and automated sync in ArgoCD?

Manual sync requires explicit approval before applying changes; automated sync applies changes automatically on Git commits.

Q3: Why is the "prune" policy important?

Prune deletes resources that were removed from Git, preventing orphaned resources from cluttering the cluster.

Q4: What does "self-heal" do in ArgoCD?

Self-heal automatically reverts manual cluster changes and re-syncs to Git state, enforcing desired state.

Q5: When would you use the App-of-Apps pattern?

To manage multiple applications in a single cluster, or deploy to multiple clusters/environments from one root Application.
