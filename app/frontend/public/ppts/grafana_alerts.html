Grafana Alerting Architecture

Alert State Machine:

Alert Rules → Evaluation ↓ State: Normal | Pending | Firing ↓ Contact Points (Slack, PagerDuty, Email) ↓ Notifications Sent

Components:

  • Alert Rules: Query + threshold conditions
  • Notification Policies: Routing & grouping rules
  • Contact Points: Destinations (integrations)
  • Silences: Suppress alerts during maintenance

Creating Alert Rules

Alert Rule Composition:

  • Query: From Prometheus/Loki/etc
  • Condition: Threshold (>, <, ==)
  • Evaluate for: How long before firing (default: 5m)
  • Labels: Severity, team, service tags
  • Annotations: Summary, description, runbook URL

Example: Alert if CPU > 85% for 5+ minutes

Query: avg(cpu_usage) > 85 For: 5m Label: severity=warning Annotation: runbook=https://wiki/cpu-high

Contact Points

Supported Integrations:

  • Slack: #channel, @user mentions, rich formatting
  • PagerDuty: Create incidents, escalation policies
  • Email: SMTP configuration required
  • OpsGenie: Incident management
  • Webhook: Custom integrations (Jira, custom APIs)

Contact Point Config: Name, type, credentials (API keys, tokens)

Sensitive data stored in Grafana secrets backend

Notification Policies

Routing Tree: Root policy + nested matchers

Root (default contact point): ├─ Matcher: severity=critical │ └─ Contact: PagerDuty └─ Matcher: team=database └─ Contact: DBA Slack channel

Grouping: Combine related alerts

  • Group by: severity, team, service
  • Group wait: 10s (wait before sending first)
  • Repeat interval: 4h (resend resolved alerts)

Silence & Inhibition

Silence: Suppress all matching alerts temporarily

  • During maintenance windows
  • Time-bound (start/end time)
  • Matcher: Labels to match (team, service)

Inhibition Rules: Suppress alerts conditionally

Inhibit if: severity=critical Matches: severity=warning # Suppress warnings when critical alert exists

On-Call Integration

PagerDuty Integration:

Quiz: Grafana Alerts & Integrations

1. What are the three states an alert can have?

Normal - Condition is not met. Pending - Condition met but "for" duration not reached. Firing - Condition met for "for" duration, alert is sent.

2. When would you use an Inhibition Rule?

To prevent alert noise - e.g., suppress "disk space low" warnings when there's a "disk full" critical alert. Avoids duplicate notifications about the same underlying issue.

3. What is the difference between Silence and Inhibition?

Silence: Temporary, manual suppression during maintenance (time-bound). Inhibition: Permanent rule-based suppression of dependent alerts (e.g., critical suppresses warning).

4. How does notification grouping reduce alert fatigue?

Grouping combines multiple alerts with the same labels (e.g., all pod restart alerts) into a single notification. This reduces the number of messages sent to Slack/PagerDuty, avoiding alert storms.

5. What information should an alert annotation include?

Summary: What triggered. Description: Current values and context. Runbook URL: How to resolve. Dashboard link: For investigation. These help on-call engineers respond quickly.
