Segment 1 — Why Capacity

Why Capacity Planning Matters

Avoid Outages

  • › Resource exhaustion = cascading failures
  • › OOMKilled pods, CPU throttling, node disk pressure
  • › Sudden spike hits zero headroom → crash
  • › Plan ahead, not during an incident

Cost + Growth

  • › Over-provisioning = wasted spend
  • › Under-provisioning = poor UX, SLO misses
  • › SLOs require guaranteed capacity
  • › Forecast business growth → right-size infrastructure
Segment 2 — Demand Forecasting

Demand Forecasting

Forecasting Methods

  • › Historical trends — analyze past 6-12 months
  • › Seasonal patterns — holidays, weekends, time-of-day
  • › Business spikes — product launches, sales events
  • › Growth curves — linear vs exponential
  • › Safety margin — add 20-30% buffer to forecast

Example

# Forecast: 2M req/day
# Peak: 200k req/sec (100x avg)
# Apply 25% safety margin:
Peak forecast: 250k req/sec

# If 1 pod = 50k req/sec
Replicas needed: 250k / 50k = 5
Segment 3 — Load Modeling

Load Modeling: Requests → Nodes

The Formula

Requests/sec → CPU/Memory/Network per pod → Replica count → Node count

Load Testing

  • › Run throughput test with 1 pod
  • › Find: "max req/sec before p99 > SLO"
  • › Measure CPU, memory, network usage
  • › Example: 1 pod = 50k req/sec @ 2 CPU, 512 MiB

Node Sizing

  • › Total pod CPU → pod count → node count
  • › Example: 250k req/sec ÷ 50k = 5 pods
  • › 5 pods × 2 CPU = 10 CPU total
  • › Use 4-CPU nodes, need 3 nodes (leave room for k8s)
Segment 4 — K8s Capacity

K8s Capacity Planning

Resource Quotas

  • › Namespace quotas — cap requests/limits per namespace
  • › LimitRanges — enforce min/max per pod or container
  • › Pod request — used for scheduling, not actual limit
  • › Pod limit — maximum allowed, pod killed if exceeded

Cluster Capacity

  • › = Sum of all node allocatable resources
  • › Minus: kubelet, system pods, reserved for node
  • › Allocatable = Capacity - Overhead
  • › Use kubectl top nodes to monitor utilization
Segment 5 — Cloud Quotas

Cloud Quotas & Limits

AWS Service Quotas

# Common limits (default, regional):
EC2 On-Demand instances: 20
Elastic IPs: 5
ENIs per instance: max_cpu_count / 2
RDS databases: 40

# → Hit limits silently, deployments fail
# → Request limit increases BEFORE you need them

Best Practices

  • › Monitor quota utilization via AWS Trusted Advisor
  • › Request increases 2-4 weeks before peak season
  • › Cross-region capacity strategy for multi-region apps
Quiz

Capacity Planning Quiz ⚡

Q1. You forecast 200k req/sec peak. With 20% safety margin, how many req/sec should you plan for?
✅ 240k req/sec (200k × 1.2). Safety margins (20-30%) protect against forecast errors and unexpected spikes.
Q2. One pod handles 50k req/sec. You need 300k req/sec. How many pods?
✅ 6 pods (300k ÷ 50k). Add 1-2 more for HPA headroom and rolling updates.
Q3. What's the difference between a Kubernetes resource request and limit?
✅ Request = scheduler guarantee, used for scheduling. Limit = hard cap; pod is evicted/killed if exceeded.
Q4. You hit an AWS EC2 On-Demand quota limit during scaling. What went wrong?
✅ You didn't proactively request a quota increase. AWS quotas are conservative by default; request increases before peak demand.
Q5. Node allocatable capacity = ?
✅ Total node capacity minus kubelet, system pods, and reserved resources. Not all 4 CPUs are available to user workloads.
