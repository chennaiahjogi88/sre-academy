Definitions: SLI, SLO, SLA

SLI (Service Level Indicator)

What you measure

Example: Request success rate = (successful requests / total requests) × 100%

SLO (Service Level Objective)

What you target

Example: Maintain 99.9% success rate SLI over rolling 30-day window

SLA (Service Level Agreement)

Contractual commitment with external parties, includes consequences for breaches

Example: Guarantee 99.9% uptime or provide service credits (typically tighter than internal SLO)

Good SLI Types

  • Availability: successful requests / total requests (most common)
  • Latency: percentage of requests with latency < threshold (p50, p95, p99)
  • Throughput: requests handled per second (capacity metric)
  • Error Rate: 4xx/5xx errors (opposite of availability, but different perspective)
  • Freshness: for data pipelines — how recent is the data being served?

Golden Rule: SLIs should correlate with user happiness. If users are happy, your SLIs are good.

Setting SLOs: Start with Users

The Process

  • Start with user pain points: "What makes users unhappy?"
  • Work backwards: At what latency/error rate do users complain?
  • Don't set SLOs too tight initially — room to improve later
  • SLO should be harder than current state but achievable

SLO Document Format

  • Service: which service?
  • SLI Query: exact metric (PromQL, CloudWatch, etc.)
  • SLO Target: e.g., 99.9%
  • Measurement Window: rolling 30 days, calendar month, etc.

SLA vs SLO vs SLI

SLI (Measurement)
↑
SLO (Internal Target)
↑
SLA (External Promise)

Rule: SLA < SLO — under-promise, over-deliver
Example: SLI is 99.95%, SLO is 99.9%, SLA is 99%

Quiz: SLI, SLO & SLA

Q1: If your SLO is 99.9%, how many minutes of downtime are you allowed per 30 days?
A: 43.8 minutes. Calculation: 30 days × 24 hours × 60 min × (1 - 0.999) = 43.8 minutes
Q2: Which is tighter: SLA or SLO?
A: SLO (internal target) should be tighter than SLA (external promise). Example: SLO 99.9%, SLA 99%
Q3: What does SLI stand for?
A: Service Level Indicator — the actual metric you measure (e.g., request success rate)
Q4: Name three good SLI types for a web service.
A: Availability (success rate), Latency (p95/p99 < threshold), Error rate (% of 4xx/5xx responses)
Q5: Why should SLOs correlate with user happiness?
A: SLOs should reflect what users actually care about. If SLO looks good but users are unhappy, you're measuring the wrong thing.
