Why Terraform?

Declarative IaC: Describe desired state, Terraform handles how
Multi-Cloud: AWS, Azure, GCP, on-prem all supported
State Management: Tracks infrastructure state for safe updates
Plan Before Apply: Preview changes before executing
GitOps-Friendly: Infrastructure in version control
Idempotent: Running multiple times yields same result

HCL Basics

terraform { required_version = "~> 1.5" required_providers { aws = { source = "hashicorp/aws" version = "~> 5.0" } } } provider "aws" { region = var.aws_region } resource "aws_vpc" "main" { cidr_block = "10.0.0.0/16" enable_dns_hostnames = true } output "vpc_id" { value = aws_vpc.main.id description = "VPC ID" }

Core Terraform Commands

terraform init

Initialize working directory, download provider plugins

terraform plan

Show what will be created/changed/destroyed

terraform apply

Apply planned changes to infrastructure

terraform destroy

Destroy all managed infrastructure

terraform validate

Check syntax and config validity

terraform state list

List all resources in state file

State Management

Local vs Remote State

Local (terraform.tfstate): Simple, risky for teams, prone to corruption

Remote State (S3 + DynamoDB): Shared, locked, auditable, encrypted

terraform { backend "s3" { bucket = "my-tf-state" key = "prod/terraform.tfstate" region = "us-east-1" encrypt = true dynamodb_table = "tf-locks" } }

Always commit state lock requirements, never commit state files to git

Knowledge Check

Q1: What is the purpose of terraform.tfstate?

Stores the actual state of provisioned infrastructure, enabling Terraform to track and manage resources.

Q2: What command shows changes before applying them?

terraform plan — displays what will be created, modified, or destroyed.

Q3: Why use S3 + DynamoDB for state locking?

DynamoDB provides distributed locking to prevent concurrent state modifications by multiple teams.

Q4: What is idempotency in Terraform?

Running Terraform multiple times with same config yields the same result — safe to re-apply.

Q5: When would you use modules?

To organize, reuse, and version infrastructure components across projects and teams.
