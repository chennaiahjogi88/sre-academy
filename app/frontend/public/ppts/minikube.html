Segment 1 — Setup

Installing Minikube

📦 Prerequisites

  • CPU: 2 cores minimum
  • RAM: 2GB minimum (4GB recommended)
  • Driver: Docker, VirtualBox, or Hyperkit
  • kubectl: Install separately

⚙️ Install Commands

# macOS brew install minikube # Linux curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64 sudo install minikube-linux-amd64 /usr/local/bin/minikube # Start cluster minikube start --cpus=2 --memory=4096 --driver=docker
✅ After start: kubectl get nodes should show one node with status Ready. kubectl cluster-info confirms API server is running.
Segment 2 — kubectl Lab

Essential kubectl Commands

# Deploy & manage kubectl create deployment nginx --image=nginx:alpine --replicas=3 kubectl expose deployment nginx --type=NodePort --port=80 minikube service nginx --url # get access URL # Inspect kubectl get all # everything in default namespace kubectl describe pod <pod-name> # detailed pod info + events kubectl logs <pod-name> -f # follow logs kubectl exec -it <pod-name> -- sh # shell into container # Scale & update kubectl scale deployment nginx --replicas=5 kubectl set image deployment/nginx nginx=nginx:1.25 # rolling update kubectl rollout status deployment/nginx kubectl rollout undo deployment/nginx # rollback # Cleanup kubectl delete deployment nginx kubectl delete service nginx

Minikube Addons

Segment 4 — Lab

Deploy a Full Stack App

# 1. Create namespace kubectl create namespace demo # 2. Deploy Postgres kubectl create deployment postgres --image=postgres:15 -n demo kubectl set env deployment/postgres POSTGRES_PASSWORD=secret -n demo # 3. Deploy your API kubectl create deployment api --image=myapi:v1 -n demo --replicas=2 # 4. Expose with ClusterIP (internal) kubectl expose deployment postgres --port=5432 -n demo kubectl expose deployment api --port=8080 --type=NodePort -n demo # 5. Apply HPA (requires metrics-server addon) kubectl autoscale deployment api --cpu-percent=70 --min=2 --max=10 -n demo # 6. Check everything kubectl get all -n demo kubectl top pods -n demo
Segment 5 — Ingress

Ingress & Service Tunneling

Enable Ingress

# Enable ingress addon minikube addons enable ingress # Create a basic Ingress apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: demo-ingress spec: rules: - host: demo.local http: paths: - path: / pathType: Prefix backend: service: name: api port: {number: 8080}

Access Services Locally

# Add to /etc/hosts for ingress echo "$(minikube ip) demo.local" | \ sudo tee -a /etc/hosts # Or use minikube tunnel (LoadBalancer) minikube tunnel # Then service gets external IP 127.0.0.1 # Or port-forward directly kubectl port-forward svc/api 8080:8080 # Open service in browser minikube service api --url
Segment 6 — Debugging

Debugging Pods & Services

Pod Not Starting?

# Check pod status and events kubectl describe pod <pod-name> # Look for: Events at the bottom # Common: ImagePullBackOff, CrashLoopBackOff # Check logs kubectl logs <pod-name> kubectl logs <pod-name> --previous # last crash # Exec into running pod kubectl exec -it <pod-name> -- sh # Run a debug pod with tools kubectl run debug --image=nicolaka/netshoot \ -it --rm -- bash

Service Not Reachable?

# Check endpoints (are pods selected?) kubectl get endpoints <service-name> # If NONE: label selector mismatch # Check service definition kubectl describe svc <service-name> # Test DNS resolution from a pod kubectl run test --image=busybox -it --rm \ -- nslookup <service-name> # Check minikube dashboard minikube dashboard
Knowledge Check

Quick Quiz 🎯

1. What command shows logs for a pod and keeps following new output?
kubectl logs <pod-name> -f (the -f flag means "follow", like tail -f)
2. Which addon must be enabled for HPA to work in Minikube?
metrics-server — minikube addons enable metrics-server. Without it, kubectl top and HPA have no CPU/memory data.
3. How do you get the URL to access a NodePort service in Minikube?
minikube service <service-name> --url — Minikube maps the NodePort to a reachable localhost URL since the node is a VM/container.
4. You deployed v1 of your app. After updating to v2 there's a bug. How do you rollback?
kubectl rollout undo deployment/<name> — rolls back to the previous ReplicaSet. Check history with kubectl rollout history deployment/<name>.
