Segment 1 — Fundamentals

Three Storage Paradigms

Block Storage

  • Raw volumes attached to instances
  • OS formats with filesystem (ext4, xfs)
  • Low latency, high IOPS
  • Single-instance attachment (default)
  • AWS: EBS

Object Storage

  • Flat namespace — key → value
  • No filesystem — HTTP API (PUT/GET)
  • Virtually unlimited capacity
  • Ideal for static assets, backups, data lake
  • AWS: S3

File Storage

  • NFS/SMB — shared directory tree
  • Multiple instances mount same volume
  • POSIX-compatible filesystem
  • Shared config, ML datasets, CMS content
  • AWS: EFS
💡 Rule of thumb: EBS for your OS/app data · S3 for artifacts & media · EFS for shared workloads
Segment 2 — S3

Amazon S3 — Core Concepts

Buckets & Objects

  • Bucket = globally unique namespace container
  • Object = file + metadata, up to 5 TB
  • Key = full path (e.g. logs/2025/app.log)
  • Multipart upload for objects > 100 MB
  • 11 nines (99.999999999%) durability
  • Access via console, CLI, SDK, or presigned URL

Key Features

  • Versioning — keep every version of an object
  • Replication — CRR (cross-region), SRR (same-region)
  • Lifecycle rules — auto-transition & expire objects
  • Server-side encryption — SSE-S3, SSE-KMS, SSE-C
  • Static website hosting — bucket as web origin
  • Event notifications → Lambda/SQS/SNS
Block Public Access — on by default Bucket Policy — JSON resource-based policy ACLs — legacy, prefer bucket policies

Segment 3 — EBS

Amazon EBS — Elastic Block Store

Volume Types

  • gp3 — General purpose SSD, 3,000 IOPS baseline, up to 16,000 IOPS independently
  • io2/io2 Block Express — Provisioned IOPS SSD, up to 256,000 IOPS (databases)
  • st1 — Throughput optimized HDD, big data & log processing
  • sc1 — Cold HDD, lowest cost archival workloads

Key Properties

  • Attached to one EC2 instance (in same AZ)
  • io1/io2 supports Multi-Attach (up to 16 instances)
  • Snapshots — incremental, stored in S3
  • Snapshots are cross-region copyable
  • Encryption at rest using KMS (AES-256)
  • Resize & change type without downtime (elastic volumes)
💡 Root volume is EBS by default. Snapshots enable DR — automate with AWS Backup or DLM (Data Lifecycle Manager)
Segment 3 — EFS

Amazon EFS — Elastic File System

What It Is

  • Fully managed NFS v4.1/4.2 file system
  • Mount across multiple EC2 instances simultaneously
  • Scales automatically — pay per GB used
  • Available within a VPC via mount targets (one per AZ)
  • Also accessible from on-prem via Direct Connect / VPN
  • POSIX-compliant — works with Linux workloads

Storage Classes & Performance

  • EFS Standard — frequently accessed data
  • EFS Standard-IA — infrequent access, auto-lifecycle
  • Bursting mode — throughput scales with size
  • Provisioned mode — set throughput independently
  • Elastic mode — auto-scales throughput (recommended)
  • Encryption in transit (TLS) & at rest (KMS)
Kubernetes PersistentVolumes via EFS CSI Driver Shared home dirs, CMS assets, ML datasets
Segment 4 — RDS

Amazon RDS — Relational Database Service

Managed for You

  • Engines: MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, Aurora
  • AWS handles: patching, backups, failover, monitoring
  • Automated backups — point-in-time recovery (up to 35 days)
  • Manual snapshots — retained until deleted
  • Storage autoscaling — grows automatically
  • Multi-AZ — synchronous standby replica for HA

Read Replicas & Aurora

  • Read Replicas — async replication, up to 15 per source, cross-region supported
  • Aurora — MySQL/PostgreSQL compatible, 5× faster, 6-copy storage across 3 AZs
  • Aurora Serverless v2 — auto-scales capacity in fine-grained ACUs
  • Global Database — <1 second replication across regions
  • Proxy — connection pooling, failover in <30s
💡 Multi-AZ = HA (same region, automatic failover). Read Replica = read scaling. They are different things.
Segment 5 — DynamoDB

Amazon DynamoDB

NoSQL at Any Scale

  • Fully managed key-value & document store
  • Single-digit millisecond performance at any scale
  • No servers to manage — serverless
  • Table → Items (rows) → Attributes (columns)
  • Partition Key — uniquely identifies item (hash key)
  • Sort Key — optional, enables range queries
  • Strong or eventual consistency reads

Key Features

  • On-Demand — pay per request, no capacity planning
  • Provisioned — set RCU/WCU, use Auto Scaling
  • DynamoDB Streams — ordered change log → Lambda triggers
  • Global Tables — multi-region active-active replication
  • DAX — in-memory cache, microsecond reads
  • TTL — auto-expire items, no extra cost
  • GSI/LSI — Global/Local Secondary Indexes for query flexibility

  • aws s3 mb s3://my-bucket — create bucket
  • aws s3 cp file.txt s3://my-bucket/ — upload
  • aws s3 sync ./dir s3://my-bucket/ — sync dir

Quiz

Storage & Database Quiz ⚡

Q1. Which S3 storage class provides the lowest cost for data accessed only once a year?
✅ Glacier Deep Archive — cheapest storage class, designed for 7-10 year retention, 12-hour retrieval time.
Q2. Can you mount an EBS gp3 volume on two EC2 instances simultaneously?
✅ No (default). EBS volumes attach to one instance. Use io1/io2 Multi-Attach for shared block storage, or use EFS for NFS-based shared access.
Q3. What's the difference between RDS Multi-AZ and a Read Replica?
✅ Multi-AZ is for High Availability (synchronous standby, automatic failover, same region). Read Replicas are for read scaling (async replication, can be cross-region, you must point app to the replica endpoint).
Q4. In DynamoDB, what is the Partition Key used for?
✅ The Partition Key (hash key) determines which partition holds the item. It must be unique if used alone, or combined with a Sort Key to form a composite primary key.
Q5. You need shared storage for 10 EC2 instances running a web app. Which AWS service?
✅ Amazon EFS — it supports NFS mounts from multiple EC2 instances simultaneously. S3 could work for assets but isn't a POSIX filesystem. EBS can't be shared across instances (without Multi-Attach).
