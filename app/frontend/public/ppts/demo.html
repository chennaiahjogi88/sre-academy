What is SRE?

🏛️ Origin Story

Google invented SRE in early 2000s, led by software engineers tasked with solving operational reliability at scale.

📖 Definition

SRE is the practice of applying software engineering principles to IT operations for highly reliable, scalable services.

🎯 Mission

SREs protect, provide for, and progress software systems (Gmail, YouTube, Ads, Search) with high availability and performance.

Core Approach: SRE teams use coding skills to automate tasks, manage incidents, and constantly improve reliability.

📊 Understanding SLIs, SLOs, and SLAs

🎯 Service Level Indicator (SLI) - What you measure

Metrics that quantify service health

  • Examples: Latency, error rate, availability, throughput
  • Sample: "99.5% of requests completed in < 200ms"

🎯 Service Level Objective (SLO) - Your internal target

Target values for SLIs that set reliability expectations

  • Sample: "99.9% uptime over 30 days"

📜 Service Level Agreement (SLA) - Your promise to customers

Legal/business commitment with consequences

  • Sample: "99.5% uptime guarantee or customer gets credit"

Formula:

SLA ≤ SLO - Buffer ≤ Current Performance
99.5% ≤ 99.9% - 0.4% ≤ 99.95%

💰 Error Budgets: Balancing Speed and Stability

What is an Error Budget?

The acceptable amount of unreliability defined by your SLO

📐 Calculation Example:

If SLO = 99.9% availability over 30 days:

  • Total minutes in 30 days: 43,200 minutes
  • Allowed downtime (0.1%): 43.2 minutes
  • Error Budget = 43 minutes per month
Scenario Action
Error budget remaining Push new features aggressively
Error budget depleted Freeze deployments, focus on stability
50% budget used mid-month ⚠️ Slow down, increase testing
Data-driven decisions > Opinions

⚡ SRE Principles

1. Embrace Risk

Accept that incidents will happen; manage with error budgets

"100% reliability is impossible and unnecessary"

2. Set SLOs/SLIs/SLAs

Explicitly define what reliability means

"If you can't measure it, you can't improve it"

3. Eliminate Toil

Focus on automating manual/repetitive work

Target: < 50% time on toil

4. Effective Monitoring

Setup proactive, complete visibility into system

"Detect problems before users do"

5. Simplicity

Favor clear, maintainable systems

"Complexity is the enemy of reliability"

6. Blameless Postmortems

Learn from incidents without finger-pointing

"Systems fail, not people"

7. Continuous Improvement

Use feedback loops, strive for incremental reliability improvements

"Every incident is a learning opportunity"

🔧 Understanding Toil

Definition

Toil is manual, repetitive, automatable work that scales linearly with service growth

Characteristics of Toil:

  • Manual (requires human intervention)
  • Repetitive (same task over and over)
  • Automatable (can be scripted/coded)
  • No enduring value (temporary fix)
  • Scales with service (more users = more toil)

Better Approaches:

  • Auto-restart with health checks
  • Auto-scaling based on metrics
  • Self-service password reset
  • CI/CD pipeline automation
  • Fix root cause, not symptoms

🎯 SRE Target:

Spend < 50% time on toil, > 50% on engineering work

🔄 SRE vs. DevOps

Aspect DevOps SRE
Focus Cultural movement, collaboration Specific implementation with engineering focus
Approach Break down silos between Dev & Ops Software engineering applied to operations
Primary Goal Faster delivery, continuous deployment Reliability, availability, scalability
Metrics Deployment frequency, lead time SLOs, SLIs, error budgets, MTTR
Responsibility Shared between Dev & Ops Dedicated SRE team with clear ownership
Decision Making Consensus-based Data-driven (error budgets)

💡 Key Insight:

SRE can be seen as a specific implementation of DevOps principles with an engineering-first approach

"SRE is what happens when you ask a software engineer to design an operations team" - Google

👨‍💻 SRE in Practice - Real-world Responsibilities

1. 📊 Monitoring and Observability

Design dashboards, set up alerting systems

Tools: Prometheus, Grafana, ELK, Datadog | Example: Creating dashboards for API latency, error rates

2. 🚨 Incident Response

On-call support, manage postmortems after outages, follow incident severity protocols (SEV1/SEV2/SEV3)

Example: Responding to production database failure at 3 AM

3. 📈 Capacity Planning

Ensure service can scale with demand, forecast resource needs based on growth trends

Example: Planning infrastructure for Diwali sale traffic

4. 🤖 Automation

Script deployments, automate health checks and recovery, build self-healing systems

Example: Auto-scaling groups, auto-restart on failures

5. 🤝 Collaboration

Work with developers to design reliable features, review code for stability, participate in architecture discussions

Example: Code reviews focusing on error handling, retry logic

🚀 SRE Career Path

Junior SRE (0-2 years)

  • Learn monitoring tools, scripting
  • Participate in on-call rotation with mentorship
  • Work on small automation tasks
  • Skills: Linux, Python/Go, basic cloud (AWS/GCP/Azure)

SRE (2-4 years)

  • Own specific services/systems
  • Lead incident responses
  • Skills: Kubernetes, Terraform, CI/CD, monitoring stack
