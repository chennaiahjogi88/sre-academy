Segment 1 — ConfigMaps

ConfigMaps — Externalize Config

📄 Create a ConfigMap

apiVersion: v1 kind: ConfigMap metadata: name: app-config data: APP_ENV: production LOG_LEVEL: info DB_HOST: postgres.default.svc

🔗 Use in Pod

envFrom: - configMapRef: name: app-config # OR mount as a file volume: volumeMounts: - name: config mountPath: /etc/config
💡 Best practice: Use ConfigMaps for non-sensitive config (URLs, feature flags, log levels). Update ConfigMaps without redeploying by mounting as volumes (requires pod restart to see changes with envFrom).
Segment 2 — Secrets

Kubernetes Secrets

🔐 What Are Secrets?

  • Store sensitive data: passwords, tokens, certs
  • Base64 encoded (NOT encrypted by default!)
  • Stored in etcd — enable encryption at rest
  • Types: Opaque, TLS, docker-registry
  • Mounted as env vars or volumes

⚙️ Secret YAML

kind: Secret metadata: name: db-secret type: Opaque stringData: # auto base64 DB_PASSWORD: s3cr3t! API_KEY: abc123xyz
⚠️ Production rule: Use external secret managers (AWS Secrets Manager, Vault) + External Secrets Operator or Sealed Secrets. Never commit plain Secrets to Git.
Segment 3 — Controllers

DaemonSets & StatefulSets

👾 DaemonSet

  • Runs one pod per node automatically
  • New nodes get the pod automatically
  • Use for: log collectors, metrics agents, CNI plugins
  • Examples: Fluentd, Node Exporter, Datadog Agent
  • Can use nodeSelector to limit nodes

🗄️ StatefulSet

  • Pods get stable, ordered names (pod-0, pod-1)
  • Each pod gets its own PersistentVolume
  • Ordered startup and shutdown
  • Stable network identity (DNS per pod)
  • Use for: Postgres, Kafka, Zookeeper, etcd
🔑 Key difference: Deployment pods are interchangeable (cattle). StatefulSet pods have identity (pets). Use Deployments for stateless apps, StatefulSets for databases.
Segment 4 — Jobs

Jobs & CronJobs

⚡ Job

  • Runs pod(s) to completion, then stops
  • Retries on failure (backoffLimit)
  • Supports parallelism for batch processing
  • Pod stays for log inspection after completion
  • Use for: DB migrations, one-off scripts

🕐 CronJob

kind: CronJob spec: schedule: "0 2 * * *" # 2am daily jobTemplate: spec: template: spec: containers: - name: backup image: backup-tool:v1
Knowledge Check

Quick Quiz 🎯

1. You need to run a Prometheus node-exporter on every node in the cluster. What controller do you use?
DaemonSet — automatically runs exactly one pod per node. New nodes automatically get the pod without any manual intervention.
2. Why are Secrets base64 encoded instead of encrypted?
Base64 is just encoding, not encryption — it's for safe transport of binary data. Secrets are NOT encrypted by default in etcd. You must enable encryption at rest separately, or use an external secret manager.
3. A StatefulSet has 3 replicas. In what order do pods start?
Ordered: pod-0 first (must be Running & Ready), then pod-1, then pod-2. Shutdown is reverse order: pod-2, pod-1, pod-0.
4. What happens if a CronJob's previous run hasn't finished when the next schedule triggers?
Depends on concurrencyPolicy: Allow (run both), Forbid (skip new), Replace (cancel old, start new). Default is Allow.
