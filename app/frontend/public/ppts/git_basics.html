Git Core Concepts

# Git workflow: Working Tree → Staging → Local Repo → Remote $ git add file.txt # Stage changes $ git commit -m "Fix bug" # Commit locally $ git push origin main # Push to remote # Configure Git $ git config --global user.name "Your Name" $ git config --global user.email "email@example.com" # .gitignore example *.log node_modules/ .env terraform.tfstate

Key areas: Working tree (files on disk) → Staging area (to commit) → Local repo (history) → Remote (GitHub/GitLab)

Branching Strategies

GitHub Flow

Simple: main + feature branches. PR → review → merge. Best for continuous deployment.

Gitflow

Complex: main/develop/feature/release/hotfix. Good for planned releases, more overhead.

Trunk-Based Dev

Minimal branches, PRs merge to main daily. Requires feature flags, high discipline.

Feature Branches

Branch per feature: feature/auth, feature/api-v2. Clear intent, easy rollback.

Merge vs Rebase vs Squash

Merge (Fast-Forward or Create Merge Commit)

Use: Preserves full history, clear when feature branch merged

git merge feature-branch

Rebase (Rewrite History)

Use: Linear history, cleaner log, easier to understand flow

git rebase main

Squash (Combine All Commits)

Use: Clean single commit per feature, hides intermediate work

git merge --squash feature-branch

Summary of changes ## Why? Problem being solved ## How to test? Steps to verify ## Checklist - [ ] Tests added - [ ] Docs updated - [ ] No breaking changes

Trunk-Based Development

Short-Lived Branches: Merge to main within 1 day, no long-running branches
Feature Flags: Deploy incomplete features hidden behind flags
Continuous Integration: Every commit tested, frequent merges reduce conflicts
High Discipline: Requires excellent testing and monitoring
Benefits: Faster releases, fewer merge conflicts, easier rollbacks
Tools: LaunchDarkly, Flagsmith, or custom feature flag service

Advanced Git Commands

git stash

Temporarily save uncommitted changes without committing

git cherry-pick

Apply specific commit from another branch to current

git bisect

Binary search to find which commit introduced a bug

git reflog

Recover lost commits, undo resets

git blame

See which commit changed each line

git log --graph

Visualize branch history and merges

Knowledge Check

Q1: What is the difference between git merge and git rebase?

Merge preserves full history with merge commits; rebase rewrites history linearly. Use merge for shared branches, rebase for local cleanup.

Q2: Why is trunk-based development preferred in SRE?

Short-lived branches reduce merge conflicts, enable frequent releases, and simplify CI/CD pipelines for rapid iteration.

Q3: What should always be in .gitignore?

Secrets (.env), terraform.tfstate, logs (*.log), dependencies (node_modules/), and any sensitive data.

Q4: When would you use git stash?

To temporarily save uncommitted work when switching branches without committing half-done changes.

Q5: What is a draft PR and why use it?

A PR marked as draft won't merge until ready, good for early feedback and discussing design before full implementation.
