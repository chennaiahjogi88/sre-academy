Segment 2 — Resources

CPU & Memory: Requests vs Limits

resources: requests: cpu: "250m" # 0.25 cores — guaranteed memory: "256Mi" # guaranteed RAM limits: cpu: "1" # max CPU (throttled, not killed) memory: "512Mi" # max RAM (OOMKilled if exceeded)
Request = what the scheduler reserves. Pod won't start if no node has this free.
Limit = hard cap. CPU is throttled; memory causes OOMKill if exceeded.
QoS Guaranteed = request == limit. Highest priority, last to be evicted.
QoS BestEffort = no requests/limits set. First to be evicted under pressure.
Segment 3 — Affinity

Node Affinity & Node Selector

🏷️ nodeSelector (simple)

nodeSelector: disktype: ssd zone: us-east-1a

Simple key=value matching. Hard requirement — pod won't schedule if no match.

🎯 Node Affinity (advanced)

affinity: nodeAffinity: requiredDuringSchedulingIgnoredDuringExecution: nodeSelectorTerms: - matchExpressions: - key: gpu operator: In values: [true]
📌 Use preferredDuringScheduling for soft rules (try but don't fail) vs requiredDuringScheduling for hard rules
Segment 4 — Taints

Taints & Tolerations

🚫 Taints (on Nodes)

# Add a taint to a node kubectl taint nodes node1 gpu=true:NoSchedule # Effects: NoSchedule # Don't schedule without toleration PreferNoSchedule # Avoid if possible NoExecute # Evict existing pods too

✅ Tolerations (on Pods)

tolerations: - key: "gpu" operator: "Equal" value: "true" effect: "NoSchedule"

Pod can tolerate the taint — scheduler will now consider this node.

🎯 Use case: Dedicate GPU nodes for ML workloads. Taint GPU nodes, add toleration only to ML pods. Other pods won't land there.
Segment 5 — Priority

Pod Priority & Preemption

PriorityClass

apiVersion: scheduling.k8s.io/v1 kind: PriorityClass metadata: name: high-priority value: 1000000 globalDefault: false description: "Critical workloads" --- # Reference in a Pod/Deployment: spec: priorityClassName: high-priority

How Preemption Works

  • › High-priority pod can't be scheduled → scheduler looks for nodes where evicting lower-priority pods would make room
  • › Lower-priority pods are evicted (gracefully)
  • › Built-in system classes: system-cluster-critical, system-node-critical
  • › Protect critical pods: kube-dns, CNI, monitoring agents
  • ⚠ Preemption respects PodDisruptionBudgets
Segment 6 — Spread

Topology Spread Constraints

What & Why

  • › Controls how pods spread across failure domains (zones, nodes)
  • › Prevents all replicas landing in one AZ (single point of failure)
  • › More powerful than podAntiAffinity — handles uneven distributions
  • › maxSkew: max allowed difference between zones
  • › whenUnsatisfiable: DoNotSchedule or ScheduleAnyway

YAML Example

topologySpreadConstraints: - maxSkew: 1 topologyKey: topology.kubernetes.io/zone whenUnsatisfiable: DoNotSchedule labelSelector: matchLabels: app: myapp - maxSkew: 1 topologyKey: kubernetes.io/hostname whenUnsatisfiable: ScheduleAnyway
🎯 Best practice: Spread across zones (DoNotSchedule) AND nodes (ScheduleAnyway) for HA without hard blocking
Knowledge Check

Quick Quiz 🎯

1. A pod has requests: cpu=500m, memory=1Gi and limits: cpu=500m, memory=1Gi. What QoS class is it?
Guaranteed — requests equal limits. This pod is the last to be evicted under memory pressure.
2. What happens when a container exceeds its memory limit?
It gets OOMKilled (Out Of Memory killed) by the kernel. The pod may restart depending on its restartPolicy.
3. You want pods to prefer nodes with SSD but still schedule on HDD if needed. What do you use?
Node Affinity with preferredDuringSchedulingIgnoredDuringExecution. This is a soft rule — scheduler prefers SSD nodes but won't fail if unavailable.
4. You taint a node with NoExecute. What happens to pods already running on it?
They get evicted immediately unless they have a matching toleration. NoExecute affects both new scheduling and existing pods.
