Segment 1 — RBAC

Role-Based Access Control

RBAC Objects

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: default
rules:
- apiGroups: [""]
  resources: [pods, pods/logs]
  verbs: [get, list]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: pod-reader
subjects:
- kind: ServiceAccount
  name: app-sa

RBAC Best Practices

  • › Principle of least privilege — grant minimum permissions needed
  • › ServiceAccount per app — don't reuse default SA
  • › Namespace-scoped — use Role, not ClusterRole when possible
Segment 2 — Network Policies

Network Policies

NetworkPolicy Example

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
spec:
  podSelector: {}
  policyTypes:
  - Ingress
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend
spec:
  podSelector:
    matchLabels:
      app: api
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080

Network Policy Best Practices

  • › Default deny-all — start restrictive
  • › Requires CNI — Calico, Cilium, Weave
  • › Pod-to-pod isolation — cross-namespace rules too
Segment 3 — Pod Security

Pod Security Standards

Privileged

No restrictions. Rarely needed. For node components only.

Baseline

Minimal hardening. Allows most workloads. Good starting point.

Restricted

Strict security. runAsNonRoot, no root capabilities. Prod standard.

Restricted securityContext

spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    readOnlyRootFilesystem: true
    capabilities:
      drop:
      - ALL
Segment 4 — Image Security

Container Image Security

Image Best Practices

  • › Specific tags — never use "latest"
  • › Private registries — control image source
  • › Image scanning — Trivy, Snyk, ECR scanning
  • › Image signing — Cosign, Notary verify provenance

Scanning Tools

  • › Trivy — fast, comprehensive vulnerability DB
  • › Snyk — dev-friendly, container scanning
  • › ECR scanning — AWS native, auto scanning
Segment 5 — Secrets Management

Secrets Management

Secrets Anti-Patterns

  • ✗ Don't store in Git — even in private repos
  • ✗ Don't use ConfigMaps — not encrypted
  • ✗ K8s Secrets base64 ≠ encrypted — base64 is encoding, not encryption

External Secrets

  • › ExternalSecrets CRD
  • › Pull from AWS Secrets Manager
  • › Or HashiCorp Vault
  • › Auto-rotate as needed

Sealed Secrets

  • › Encrypt K8s Secrets at rest
  • › bitnami/sealed-secrets operator
  • › Safe to store encrypted in Git
Segment 6 — Audit & Readiness

K8s Audit Logging & Production Readiness

Audit Logging Setup

  • › Enable audit logging — API server logs all requests
  • › Audit policy — define what to log (Metadata, Request, RequestResponse)
  • › Shipping — send to CloudWatch, SIEM, ELK
  • › Detect anomalies — repeated access failures, privilege escalation attempts

Production Readiness Checklist

  • ✓ Resource requests/limits set
  • ✓ Probes (liveness, readiness) configured
  • ✓ PodDisruptionBudgets defined
  • ✓ RBAC minimal, ServiceAccounts per app
  • ✓ NetworkPolicies enforce isolation
  • ✓ Images pinned, scanned, signed
  • ✓ Secrets from Vault/ExternalSecrets
  • ✓ HPA configured for scaling
  • ✓ Runbooks documented, on-call ready
Quiz

K8s Security Quiz ⚡

Q1. RBAC Role grants 'pods' verb 'get'. Can you also list pods?
✅ No. You must explicitly grant 'list' verb. RBAC is deny-by-default; only granted verbs are allowed.
Q2. NetworkPolicy set to deny-all. How do you allow frontend → api traffic?
✅ Create NetworkPolicy on api pods with ingress rule from frontend pods by label selector and port.
Q3. securityContext: runAsNonRoot=true. What happens if container entry is root?
✅ Pod fails to start. K8s blocks it before execution. Forces non-root user in Dockerfile.
Q4. You store API key in K8s Secret. Is it encrypted at rest?
✅ No. K8s Secrets are base64-encoded (not encrypted). Use ExternalSecrets + Vault, or sealed-secrets.
Q5. Audit log shows 50 failed API calls from a user in 1 min. What's happening?
✅ Potential attack: brute-force, privilege escalation attempt. Alert ops, investigate user account and revoke access if needed.
