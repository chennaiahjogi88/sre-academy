Segment 1 — Alertmanager Overview

The Alert Flow

Architecture

Prometheus —[fire alert]→ Alertmanager —[dedup + group]→ Routing Tree
↓
Check: inhibit_rules? → silence? → route to receiver
↓
Send to Slack / PagerDuty / email / webhook

Key Functions

  • › Deduplication: Same alert X times → 1 notification
  • › Grouping: 50 pod-restart alerts → "Pod restart storm"
  • › Routing: Route by service/team labels
  • › Silencing: Suppress during maintenance
  • › Inhibition: Don't alert on pod restart if node is down

Why Alertmanager?

  • › Prometheus fires many duplicate alerts
  • › Each alert rule can't know all teams/channels
  • › Need intelligent grouping to reduce fatigue
  • › Need to silence in bulk during maintenance
  • › Need to correlate related alerts (suppression)
Segment 2 — Configuration

Alertmanager Configuration

alertmanager.yml Structure

global:
  resolve_timeout: 5m
  slack_api_url: https://hooks.slack.com/...

route:
  receiver: "default"
  group_by: ["severity", "service"]
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h
  routes:
  - match: { severity: "critical" }
    receiver: "pagerduty"

receivers:
  - name: "default"
    slack_configs:
    - channel: #alerts
Segment 3 — Routing Tree

Smart Alert Routing

Routing Example

# Root route (catches all)
root: receiver="default", group_by=["alertname"]
    ↓
├─ Match: {severity="critical"} → receiver="pagerduty"
├─ Match: {team="api"} → receiver="api-slack"
│ └─ Match: {service="auth"} → receiver="auth-pagerduty"
└─ Match: {team="database"} → receiver="db-slack"

Match vs Exclude

  • • match: Route if ALL labels match
  • • match_re: Regex label matching
  • • match_exclude: Route if label does NOT match
  • • Child routes inherit parent's group_by
  • • First matching route wins

Receiver Types

  • • slack_configs - Slack channels
  • • pagerduty_configs - PagerDuty incidents
  • • email_configs - Email notifications
  • • webhook_configs - Generic HTTP POST
  • • opsgenie_configs - Opsgenie on-call
Segment 4 — Grouping & Timing

Prevent Alert Storms

Grouping Timeline

# 100 pod-restart alerts fire at T=0
T=0:30 → group_wait=30s → 1 notification "50 pod restarts"
T=5:00 → group_interval=5m → "20 more pod restarts" (new window)
T=10:00 → No new alerts → silence (resolved)

# If alert fires again after 4h, repeat_interval = resend
T=4h:00 → repeat_interval=4h → resend notification

group_by

  • • Group alerts by these labels
  • • Example: [severity, service]
  • • Group(critical, api) + Group(critical, db) = 2 notifications
  • • Each group gets separate alert
  • • Prevents "one giant notification"

Timing Parameters

  • • group_wait: Wait before sending first notification
  • • group_interval: Minimum time between notifications for same group
  • • repeat_interval: How often to resend unresolved alerts
  • • Typical: 30s / 5m / 4h
Segment 5 — Inhibition & Silence

Smart Alert Suppression

Inhibition Rules

  • › Suppress child alerts when parent fires
  • › Example: Pod alerts suppressed when node is down
  • › source_match: pod_alert
  • › target_match: node_down
  • › equal: [severity, node] (both must match)
  • › Static, configured in YAML

Silences

  • › Dynamic suppression during maintenance
  • › Match alerts by regex labels + duration
  • › Use GUI: Alertmanager → Silences tab
  • › Use API: POST /api/v1/silences
  • › Use CLI: amtool silence --duration 1h
  • › Expires at end of maintenance window

Inhibition Example YAML

inhibit_rules:
- source_match:
    severity: "critical"
    alertname: "NodeDown"
  target_match:
    alertname: "PodCrashLooping"
  equal: ["node"] # Pod down suppressed if same node is down
Quiz

Alertmanager Quiz ⚡

Q1. What does Alertmanager do that Prometheus doesn't?
✅ Deduplicates, groups, and routes alerts intelligently. Prometheus just fires raw alerts — Alertmanager decides how to notify the right team.
Q2. If 100 pod-restart alerts fire in 30 seconds, how many notifications does Alertmanager send (with group_wait=30s)?
✅ One. All 100 alerts are grouped together. After 30 seconds, one notification says "100 pod restarts" instead of 100 separate messages.
Q3. What's the difference between inhibition and silence?
✅ Inhibition is static (YAML) and correlates alerts (pod alerts suppressed when node down). Silence is dynamic/temporary (GUI/API) for maintenance windows.
Q4. How would you route only critical API alerts to PagerDuty?
✅ Create a route with match: {severity: "critical", team: "api"} and receiver: "pagerduty" as a child of the root route.
Q5. What does repeat_interval do?
✅ Resends unresolved alerts after this interval (e.g., 4h). So if an alert is still firing, you get reminded every 4 hours even without new alerts.
