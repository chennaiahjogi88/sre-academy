Segment 1 — Why Load Test

Why Load Testing Matters

Find Breaking Points

  • › Discover system limits before production does
  • › Find bottlenecks: DB, API, cache, network
  • › Know max traffic your app can handle
  • › Cost: Lab testing << production outage

Validate SLOs

  • › Prove p99 latency under load
  • › Regression testing after deploys
  • › Capacity planning data
  • › Build team confidence in infrastructure
Segment 2 — Test Types

Load Test Types

Smoke

Few VUs, short duration. Baseline: system is working at all.

Load

Normal traffic. Does system handle expected user count?

Stress

Beyond capacity. Find breaking point and failure mode.

Spike

Sudden burst (10x→100x). Can autoscaling catch up?

Soak

Sustained 24+ hours. Find memory leaks, connection pool issues.

Breakpoint

Incrementally increase until failure. Exact max capacity.
Segment 3 — K6

K6: JavaScript-Based Load Testing

K6 Script Example

import http from 'k6/http';
import { check } from 'k6';

export const options = {
  vus: 50,
  duration: '5m',
  thresholds: {
    'http_req_duration': ['p(99)<200'],
    'http_req_failed': ['rate<0.01'],
  }
};

export default function () {
  const res = http.get('https://api.example.com/users');
  check(res, { 'status 200': r => r.status === 200 });
}

K6 Features

  • › Run locally: k6 run script.js
  • › Cloud distributed load: k6 cloud script.js
  • › Built-in assertions (checks, thresholds)
  • › Scenarios, ramping, gradual load increase
Segment 4 — Locust

Locust: Python-Based Testing

Locustfile Example

from locust import HttpUser, task, between

class APIUser(HttpUser):
  wait_time = between(1, 3)

  @task(3)
  def get_users(self):
    self.client.get("/users")

  @task(1)
  def create_user(self):
    self.client.post("/users", json={})

Locust Features

  • › Web UI for real-time monitoring
  • › Distributed: master + workers
  • › Python HttpUser, wait time, task weights
  • › Easy to customize user behavior
Segment 5 — Analyzing Results

Analyzing Load Test Results

Key Metrics

Latency: p50, p95, p99. Does p99 meet SLO?
Throughput: req/sec sustained. How many can system handle?
Error Rate: % 4xx/5xx. Errors under load?
Connections: Open, closed, timeouts. Connection pool exhaustion?
CPU/Memory: Monitor server during test. Any resource exhaustion?

Finding the Knee

  • › Knee of the curve — optimal throughput/latency trade-off
  • › Beyond knee — latency spikes, errors increase
  • › Set capacity at 70-80% of knee for safety margin
Quiz

Load Testing Quiz ⚡

Q1. What test type has "few VUs, short duration" and just verifies the system works?
✅ Smoke test. It's a baseline sanity check before running heavier tests.
Q2. You run a spike test (100x load) and p99 latency goes from 50ms to 2 seconds. Good or bad?
✅ Bad. Spike should cause response latency increase, but 40x is extreme. Need more nodes or code optimization.
Q3. K6 threshold: 'http_req_duration': ['p(99)<200']. What does this mean?
✅ 99th percentile latency must be under 200ms. Test fails if violated. Used for SLO validation.
Q4. In Locust, what does @task(3) on a method mean?
✅ Task weight 3 — this task runs 3x more than @task(1) tasks. Simulates realistic user behavior ratios.
Q5. You found the "knee of the curve" at 50k req/sec, p99=150ms. What capacity should you provision?
✅ ~35-40k req/sec (70-80% of knee). Leaves headroom for burst traffic and safety margin.
