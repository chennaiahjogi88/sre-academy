Observability vs Monitoring

Monitoring: Known unknowns

  • Pre-defined dashboards
  • Specific alerts you know to check

Observability: Unknown unknowns

  • Drill down into any dimension
  • Correlate signals automatically

Can you answer questions you haven't thought of yet?

The Three Pillars

  • Metrics: Numeric time-series (Prometheus, Graphite)
  • Logs: Text events with context (Loki, ELK Stack)
  • Traces: Request paths across services (Jaeger, Zipkin)

Metrics Deep Dive

Counter: Ever-increasing value (requests, errors, bytes processed)

Gauge: Can go up or down (CPU %, memory, temperature)

Histogram: Distribution across buckets (request latency)

Summary: Quantiles (p50, p95, p99 latency)

Logs Deep Dive

Structured vs Unstructured:

  • Structured: JSON with key-value pairs (queryable)
  • Unstructured: Free text (human-readable, hard to parse)

Log Levels:

  • DEBUG → INFO → WARN → ERROR

Centralized Logging: Aggregate from all services for correlation

Distributed Tracing

Trace ID: Unique identifier for entire request flow

Span: Single operation within a service

Parent Span: Relationship between spans across services

Sampling: 100% (high cost), adaptive, or error-based strategies

Essential in microservices architecture for end-to-end visibility

Observability Maturity Model

Level 1: Logs only → Limited visibility

Level 2: Metrics + Logs → Dashboard awareness

Level 3: Metrics + Logs + Traces → Full context

Level 4: Correlated signals + Automated correlation → AIOps

Quiz: Observability Foundations

1. Which of the three pillars would best help you identify that user requests are taking longer in a specific region?

Metrics - Use time-series data with geographic labels to identify latency by region. Traces could also help but metrics are more efficient for this aggregated view.

2. What is the key difference between observability and monitoring?

Monitoring answers known questions with pre-defined dashboards and alerts. Observability enables you to ask and answer new questions about your system that you hadn't anticipated.

3. Which metric type would you use to track the number of HTTP requests received?

Counter - It only increases (monotonic), perfect for counting occurrences like total requests, errors, or bytes processed.

4. In distributed tracing, what is a Trace ID used for?

A Trace ID uniquely identifies an entire request flow across all services and systems, allowing you to correlate all spans (operations) for that request.

5. Why is centralized logging important in observability?

Centralized logging aggregates logs from all services into one place, making it possible to correlate events across services and trace the full context of an issue.
