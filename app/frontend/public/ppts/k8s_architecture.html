Segment 1 — Cluster

Cluster: Control Plane + Worker Nodes

Control Plane

  • › API Server — single entry point for all requests
  • › etcd — distributed key-value store (source of truth)
  • › Scheduler — assigns pods to nodes
  • › Controller Manager — watches state, reconciles
  • › Cloud Controller Manager — cloud-specific logic

Worker Nodes

  • › kubelet — agent that runs pods on the node
  • › kube-proxy — network rules, iptables/IPVS
  • › Container Runtime — containerd, CRI-O
  • › CNI Plugin — pod networking (Calico, Cilium)
Segment 2 — Control Plane

Control Plane Components

API Server
kube-apiserver
All kubectl and internal communication goes through here. RESTful, validates and persists to etcd
etcd
Cluster State DB
Consistent distributed KV store. All cluster state lives here. Must be backed up! 3 or 5 node HA quorum
Scheduler
kube-scheduler
Watches for unscheduled pods. Scores nodes by resources, affinity, taints. Assigns pod to best node
Controller Manager
Reconciliation Loop
Runs Deployment, ReplicaSet, Node, Endpoints controllers. Watches actual vs desired state
CCM
Cloud Controller
Integrates with cloud APIs — creates LoadBalancers, PersistentVolumes, node lifecycle management
HA Setup
Multi-Master
Production clusters run 3 control plane nodes. etcd uses Raft consensus. API servers are load balanced
Segment 3 — Worker Nodes

Worker Node Components

kubelet

  • › Runs on every worker node
  • › Registers node with API Server
  • › Watches PodSpec assignments for this node
  • › Tells container runtime to start/stop containers
  • › Reports pod status back to API Server
  • › Runs liveness/readiness probes

kube-proxy

  • › Runs on every node as a DaemonSet
  • › Maintains iptables/IPVS rules for Services
  • › Translates Service virtual IPs to pod IPs
  • › Load balances across pod endpoints
  • › Some CNIs (Cilium) can replace kube-proxy
Segment 4 — Pod Lifecycle

How a Pod Gets Created

1
kubectl apply -f pod.yaml
kubectl sends HTTP POST to API Server
2
API Server validates & persists to etcd
Pod object stored in etcd with status: Pending
3
Scheduler watches, selects a node
Scores nodes, assigns pod to best fit, updates etcd
4
kubelet on node sees the pod assignment
Pulls image, creates containers via container runtime
5
Pod is Running — kubelet reports status
API Server updates etcd, kubectl get pods shows Running
Segment 5 — Namespaces

Namespaces & Essential kubectl

Namespaces

  • › Logical partitioning of cluster resources
  • › Default namespaces: default, kube-system, kube-public
  • › Resources are namespace-scoped or cluster-scoped (Nodes, PVs)
  • › Use for: env isolation (dev/staging), team isolation, RBAC
  • › Resource Quotas & LimitRanges apply per namespace

Essential kubectl Commands

  • kubectl get pods -n kube-system
  • kubectl describe pod <name> — events & status
  • kubectl logs <pod> -f — stream logs
  • kubectl exec -it <pod> -- sh — exec into pod
  • kubectl top nodes / pods — resource usage
  • kubectl get events --sort-by='.lastTimestamp'
  • kubectl rollout status deploy/<name>
Segment 6 — etcd

etcd Backup & High Availability

Why etcd Backup Matters

  • › etcd holds ALL cluster state — Pods, Deployments, Secrets, ConfigMaps
  • › Losing etcd without backup = losing the entire cluster
  • › Backup command: etcdctl snapshot save /backup/etcd.db
  • › Restore: etcdctl snapshot restore /backup/etcd.db
  • › Automate via CronJob — store snapshots in S3
  • › On managed clusters (EKS/GKE), etcd backup is managed by the cloud provider

HA Architecture

  • › etcd uses Raft consensus — needs majority (quorum)
  • › 3 nodes: quorum = 2 (can lose 1)
  • › 5 nodes: quorum = 3 (can lose 2) — recommended for large clusters
  • › Always use odd number of etcd members
  • › Multiple API servers behind a load balancer
  • › Leader election for controller-manager & scheduler
Quiz

K8s Architecture Quiz ⚡

Q1. Which component is the "source of truth" for all cluster state?
✅ etcd — a distributed key-value store where all cluster objects and their state are stored.
Q2. What decides which node a pod runs on?
✅ The kube-scheduler — it watches for unscheduled pods and assigns them to the best available node based on resources, affinity, and taints.
Q3. What is the role of the kubelet?
✅ The kubelet is an agent on every worker node. It receives pod assignments and tells the container runtime to create/stop containers, then reports status back.
Q4. How many control plane nodes do you need for HA?
✅ At least 3 — etcd uses the Raft consensus algorithm which requires a majority quorum (2 of 3) to function.
Q5. What does kube-proxy do?
✅ Maintains iptables/IPVS rules that translate Service virtual IPs to actual pod IPs, enabling pod-to-service communication.
