Segment 1 — Overview

Kubernetes Scaling Types

↔️ Horizontal (HPA)

  • More/fewer pod replicas
  • Based on CPU, memory, custom metrics
  • Best for stateless apps
  • Most common approach

↕️ Vertical (VPA)

  • Resize CPU/memory of existing pods
  • Requires pod restart to apply
  • Good for stateful, hard-to-shard apps
  • Can conflict with HPA on CPU

🖥️ Cluster (CA)

  • Add/remove nodes in node group
  • Triggered by unschedulable pods
  • Works with cloud provider APIs
  • Complements HPA
🎯 Typical pattern: HPA scales pods → pods fill nodes → CA scales nodes → new pods scheduled. They work together.
Segment 2 — HPA

Horizontal Pod Autoscaler

📄 HPA YAML

apiVersion: autoscaling/v2 kind: HorizontalPodAutoscaler spec: scaleTargetRef: kind: Deployment name: api minReplicas: 2 maxReplicas: 20 metrics: - type: Resource resource: name: cpu target: type: Utilization averageUtilization: 70

🧮 HPA Algorithm

  • Checks metrics every 15s (configurable)
  • Formula: desired = ceil(current × (current_metric / target_metric))
  • Scale-up: immediate on threshold breach
  • Scale-down: waits 5min (stabilization window)
  • Requires metrics-server to be installed
⚠️ CPU > 70% with 3 replicas → desired = ceil(3 × (90/70)) = 4

📈 Scale Out Triggers

  • Pod in Pending state due to insufficient resources
  • CA simulates scheduling to pick best node group
  • Respects min/max node group bounds

📉 Scale In Triggers

  • Node utilization < 50% for 10 min (default)
  • All pods can be moved to other nodes
  • Respects PDBs before draining
  • Skips nodes with local storage / system pods
Segment 4 — KEDA

KEDA — Event-Driven Autoscaling

⚡ What is KEDA?

  • Kubernetes-based Event Driven Autoscaler
  • Scale based on external event sources
  • Can scale to zero when no events
  • Works alongside HPA (extends it)
  • 50+ built-in scalers

📡 KEDA Scalers

  • Kafka — queue depth / consumer lag
  • RabbitMQ — message count
  • AWS SQS — queue length
  • Prometheus — any custom metric
  • Cron — time-based scaling
  • HTTP — requests per second
🎯 Use case: Scale consumer pods based on Kafka lag. 0 messages → 0 pods. 10,000 messages → 50 pods. Scales to zero saves significant compute cost.
Knowledge Check

Quick Quiz 🎯

1. HPA target CPU is 50%. Current usage is 80% with 4 replicas. How many replicas will HPA set?
ceil(4 × (80/50)) = ceil(6.4) = 7 replicas. HPA always rounds up to ensure target utilization is met.
2. Why does Cluster Autoscaler scale in slowly (10 min) but scale out quickly?
Scale-out is urgent — pending pods mean degraded service. Scale-in is conservative — evicting pods too quickly could cause disruptions. The 10-min window ensures utilization is truly low before removing nodes.
3. What is the main advantage of KEDA over HPA for queue-based workloads?
KEDA can scale to zero (no pods when queue is empty, saving cost) and scales on actual queue depth/lag rather than CPU, which is a much more meaningful metric for consumer workloads.
4. What must be installed in the cluster for HPA to work?
metrics-server — it aggregates resource metrics (CPU/memory) from kubelets and exposes them via the Kubernetes Metrics API that HPA queries.
