Segment 1 — Cost Drivers

Observability Cost Drivers

Storage

Time-series database (Prometheus/TSDB), log storage (S3, Elasticsearch), trace spans. Cost per GB/month.

Ingestion

Datadog, New Relic charge per host, per GB ingested. More data = higher cost.

Query Compute

Complex queries over large datasets. More cardinality = slower queries = higher cost.

Data Transfer

Egress from observability SaaS. Minimize by sampling and filtering at source.

Retention

Long-term storage. 30-day metrics is cheaper than 1-year retention.

Premium Features

APM, RUM, SLO management. Costs stack on top of base plan.
Segment 2 — Metric Cardinality

The Cardinality Problem

Cardinality Explosion Example

# Metric: http_requests_total
# Labels: method, path, status, user_id, request_id, pod_ip

# Unique combinations:
methods: 5 (GET, POST, PUT, DELETE, PATCH)
paths: 100 (API endpoints)
status: 10 (2xx, 3xx, 4xx, 5xx)
user_id: 1000000 (unique users)
request_id: infinite (unique per request!)
pod_ip: 100 (pods)

# Total cardinality:
5 × 100 × 10 × 1M × ∞ × 100 = DISASTER

Bad Labels

  • › user_id, request_id, session_id — unique per user/request → explosion
  • › pod_ip, container_id — high cardinality, use pod_name instead
  • › Any unbounded value — anything that grows with data
Segment 3 — Cardinality Management

Controlling Cardinality

Prometheus Strategies

  • › limit_series in scrape config — drop metrics if cardinality > threshold
  • › Recording rules — pre-aggregate high-cardinality to low-cardinality
  • › Relabeling rules — drop bad labels before storage

Grafana Tools

  • › Cardinality explorer — find worst offenders
  • › Top-N queries — which metrics cost most
  • › Label inspector — see all label values per metric
Segment 4 — Log Cost Control

Reducing Log Costs

Log Volume Reduction

Structured Logging: Only log necessary fields. Avoid verbose string output.
Log Sampling: 1% sample for INFO/DEBUG on high-traffic endpoints. 100% for errors.
Log Levels in Prod: Default INFO, not DEBUG. DEBUG only for troubleshooting.
Deduplication: Loki: dedupExpr to collapse identical logs within time window.

Loki Optimization

  • › S3 storage tiers: hot (1 day), warm (7 days), cold (archive)
  • › Retention policies per tenant, namespace, or service
  • › Bloom filters: reduce query time, lower cost
Segment 5 — Tracing Cost

Tracing Cost Optimization

Sampling Strategies

  • › Probabilistic — 1% of all traces
  • › Error-driven — 100% of error traces, 1% of success
  • › Latency-driven — Capture slow traces (p95+)
  • › Tail-based — Sample after seeing full trace

Keep What Matters

  • › All errors — 100% sampling
  • › Slow traces — p99+ latency
  • › 1% success — baseline performance
  • › Drop healthy paths — 99% silent success
Quiz

Observability Cost Quiz ⚡

Q1. Metric has labels: method, path, status, user_id. Why is user_id a problem?
✅ Unbounded cardinality. One unique value per user = millions of combinations. Use lower-cardinality labels instead (user_tier, region).
Q2. Prometheus cardinality_explorer shows 10M unique series, limit_series=1M. What happens?
✅ 9M series are dropped. Find the worst offenders (e.g., user_id label), remove them from instrumentation.
Q3. You want to keep logs but reduce cost 50%. Strategy?
✅ Log sampling: 1% of INFO logs, 100% of errors. Switch DEBUG off. Loki dedup + compression. Result: 50% volume, 90% visibility.
Q4. Tail-based tracing sampling. When is it sampled?
✅ After the complete trace arrives. Sampler sees all spans, decides if trace is worth keeping. Better decisions than head-based.
Q5. You have 1M requests/sec. Trace sampling at 1%. How many traces stored?
✅ 10k traces/sec. With 10 spans per trace = 100k spans/sec. Lower cost, still catches most issues with proper sampling strategy.
