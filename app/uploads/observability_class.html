MODULE 01

What Is Observability?

⏱ ~8 minutes · Foundation

Observability is the ability to understand the internal state of a system by examining its external outputs. It answers the question: "What is happening inside my system right now — and why?"

📘 Definition

The term comes from control theory (Rudolf Kálmán, 1960): a system is observable if, by measuring its outputs over time, you can determine its current internal state. In software, we borrow this concept — if you can determine why your system is behaving a certain way purely from its outputs (logs, metrics, traces), it is observable.

Observability vs. Monitoring — What's the Difference?

AspectMonitoringObservability
Question asked"Is the system up?""Why is it behaving this way?"
ApproachKnown failure modes → predefined alertsExplore unknown failures interactively
DataFixed dashboards, threshold alertsHigh-cardinality, correlated telemetry
AnalogyCar dashboard warning lightFull car diagnostic computer
Best forDetecting known problemsDebugging novel problems
⚠️ Common Misconception

Observability is NOT just "having dashboards." You can have hundreds of dashboards and still have a completely unobservable system if they don't let you ask arbitrary questions about your system's behavior.

The Evolution of System Understanding

1990s–2000s: Monolith Era

Simple: check CPU, memory, disk. One server, SSH in, tail the log file.

2010s: SOA / Early Microservices

Services talk over HTTP. Need centralized logging (ELK stack). APM tools emerge.

2015+: Cloud Native / Containers

Kubernetes. Ephemeral containers. Need distributed tracing. Prometheus rises.

OpenTelemetry unifies telemetry. Correlation across signals. AIOps.

MODULE 02

The Three Pillars

⏱ ~10 minutes · Core Concepts

The three pillars of observability are the foundational telemetry signals that every observable system should emit. Think of them as three lenses through which you view system behavior.

📊

Metrics

Numeric measurements aggregated over time. "What" is happening at scale.

📄

Logs

Discrete events with context. "What happened" in detail.

🔗

Traces

Request flows across services. "Where" time is spent.

🔑 Key Insight

The real power comes from correlation: a spike in a metric (Metrics) → jump to logs for that time window → link to a distributed trace to see which service caused it. The three pillars work together, not in isolation.

MODULE 03

Metrics Deep Dive

⏱ ~8 minutes · Measurements & Aggregations

Metrics are numeric time-series data. They are the most efficient telemetry signal — cheap to store, fast to query, and perfect for alerting. Every observable system needs good metrics.

The Four Golden Signals (Google SRE)

⚡
Latency

Time to serve a request. Critical: track p50, p95, p99 — not just averages. A slow 1% of requests can signal serious problems.

🚦
Traffic

Demand on the system. Requests/sec, queries/sec. Helps understand load and capacity needs.

❌
Errors

Rate of failed requests. Track both explicit (HTTP 5xx) and implicit errors (wrong results, timeouts).

📦
Saturation

How "full" the service is. CPU%, memory%, queue depth. Predicts future degradation.

Metric Types

Counter

Monotonically increasing. Total requests, total errors. Always use rate() for dashboards.

Gauge

Can go up or down. Current memory usage, active connections, queue length.

Histogram

Samples observations in buckets. Request durations, response sizes. Supports percentiles.

Summary

Pre-calculated quantiles (client-side). Less flexible than histograms for aggregation.

Prometheus — The Industry Standard

PromQL — powerful metric queries
# Error rate over 5 minutes
rate(http_requests_total{status=~"5.."}[5m]) /
rate(http_requests_total[5m]) * 100

# 99th percentile latency
histogram_quantile(0.99, 
  rate(http_request_duration_seconds_bucket[5m])
)

# Top 5 services by error rate
topk(5, 
  rate(http_requests_total{status=~"5.."}[5m])
)
✅ Best Practice

Always instrument using the USE method for resources (Utilization, Saturation, Errors) and the RED method for services (Rate, Errors, Duration). These give complete coverage without over-instrumentation.

MODULE 04

Logging Best Practices

⏱ ~8 minutes · Structured, Searchable Logs

Logs are the most familiar form of telemetry. Every developer has seen them. But there's a huge gap between having logs and having useful logs in production.

Log Levels — Use Them Correctly

LevelWhen to UseExample
TRACEExtremely detailed (dev only)Every function entry/exit
DEBUGDiagnostic informationVariable values during processing
INFONormal operational eventsService started, user logged in
WARNUnexpected but handledRetry succeeded after failure
ERRORSomething failed, needs attentionDB query failed
FATALSystem cannot continueCannot connect to required service

Unstructured vs. Structured Logging

❌ bad — unstructured log (hard to parse & search)
2024-01-15 10:23:45 ERROR Failed to process order 12345 for user john@email.com, retry 2/3
✅ good — structured JSON log (machine-readable)
{
  "timestamp": "2024-01-15T10:23:45.123Z",
  "level":     "ERROR",
  "message":  "Failed to process order",
  "service":  "order-service",
  "order_id": 12345,
  "user_id":  "usr_abc123",
  "retry":    2,
  "max_retry":3,
  "error":   "connection timeout after 30s",
  "trace_id": "abc123xyz",
  "span_id":  "def456"
}
🔑 Key Insight — Correlation ID

Always include trace_id and span_id in every log line. This is the bridge between logs and traces — you can jump from a log line directly to the full distributed trace for that request. Without this, debugging across services is nearly impossible.

Log Aggregation Architecture

Services Collector Storage Query/UI ┌──────────┐ ┌──────────┐ ┌─────────┐ ┌─────────┐ │ svc-api │──logs──► │ │ │ │ │ │ ├──────────┤ │ Fluent │──────►│ Loki / │────►│ Grafana │ │ svc-auth │──logs──► │ Bit / │ │ Elastic│ │ / Kibana│ ├──────────┤ │ Fluentd │ │ search │ │ │ │ svc-db │──logs──► │ │ │ │ │ │ └──────────┘ └──────────┘ └─────────┘ └─────────┘ parse, filter, enrich, route

What NOT to Log

  • Secrets / credentials — API keys, passwords, tokens (even masked)
  • PII without consent — full names, emails, credit card numbers, SSNs
  • High-volume noise — health check endpoints, static asset requests
  • Redundant info — don't log the same event 5 times at different layers
MODULE 05

Distributed Tracing

⏱ ~8 minutes · The Most Powerful Pillar

Distributed tracing follows a single request as it travels through multiple services. It's the pillar that makes microservices debuggable. Without it, you're guessing which of 50 services caused a slow response.

Core Concepts

Trace

A complete representation of a single request's journey. Identified by a unique Trace ID propagated through all services.

Span

A single unit of work within a trace (one service call, one DB query). Has a name, start time, duration, and tags.

Context Propagation

Passing the trace ID via HTTP headers (W3C TraceContext / B3) so all services can link their spans.

Parent Span

The span that initiated a child span. Creates the parent-child hierarchy seen in waterfall views.

Baggage

Key-value pairs propagated along with the trace. Useful for user ID, tenant ID, feature flags.

Sampling

Not every trace is recorded. Head-based vs tail-based sampling strategies control what's captured.

Implementing Tracing with OpenTelemetry

python — auto-instrumentation + manual spans
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.exporter.otlp.proto.grpc import OTLPSpanExporter

# Initialize once at startup
provider = TracerProvider()
provider.add_span_processor(
    BatchSpanProcessor(OTLPSpanExporter(endpoint="http://otel-collector:4317"))
)
trace.set_tracer_provider(provider)

tracer = trace.get_tracer(__name__)

async def process_order(order_id: str, user_id: str):
    # Create a span — becomes a child of the incoming trace
    with tracer.start_as_current_span("process_order") as span:
        span.set_attribute("order.id", order_id)
        span.set_attribute("user.id", user_id)
        
        try:
            result = await db.query("INSERT INTO orders...")
            span.set_attribute("db.rows_affected", result.rowcount)
            span.set_status(StatusCode.OK)
        except Exception as e:
            span.record_exception(e)
            span.set_status(StatusCode.ERROR, str(e))
            raise
✅ Pro Tip — Sampling Strategy

Use tail-based sampling: capture 100% of error traces and slow traces (p99+), but only 1–5% of successful fast ones. This gives you the data when you need it most without exploding storage costs.

MODULE 06

Alerting & SLOs

⏱ ~7 minutes · Acting on Signals

Data is useless without action. Alerting is how you transform observability signals into on-call incidents. But bad alerting — too noisy, too vague — leads to alert fatigue where teams ignore pages.

SLIs, SLOs, and SLAs — The Reliability Vocabulary

📏
SLI — Service Level Indicator

The actual measurement. "99.2% of requests returned success in under 300ms over the past 30 days."

🎯
SLO — Service Level Objective

The target you set. "We will achieve 99.9% availability and p99 latency < 500ms." Internal commitment.

📝
SLA — Service Level Agreement

The contract with consequences. "If availability drops below 99.5%, customers get credits." External contract.

Error Budget

An error budget is the allowable amount of unreliability derived from your SLO. If your SLO is 99.9% availability:

Error Budget Calculation SLO = 99.9% availability Period = 30 days = 43,200 minutes Allowed downtime = (1 - 0.999) × 43,200 = 43.2 minutes/month Error Budget Status: ┌─────────────────────────────────────┐ │ Budget consumed this month: 29.1m │ │ Budget remaining: 14.1m │ │ Burn rate: 67.4% │ │ Status: ● HEALTHY │ └─────────────────────────────────────┘ If burn rate > 100%, pause feature work → invest in reliability. If burn rate = 0%, you might be too conservative — ship faster!

Alert Design Principles

  • Alert on symptoms, not causes. Alert on "users are experiencing errors," not "CPU is high." High CPU is only a problem if it's causing user impact.
  • Every alert should be actionable. If an engineer receives a page and doesn't know what to do, the alert is broken.
  • Avoid alert fatigue. Noisy alerts that resolve themselves train teams to ignore pages. Aim for 0 false positives.
  • Use multi-window burn rate alerts. Fast burn (1h) catches sudden outages; slow burn (6h) catches gradual degradation.
prometheus alertmanager — error budget burn rate alert
groups:
  - name: slo_alerts
    rules:
    - alert: HighErrorBudgetBurnRate
      expr: |
        (
          rate(http_requests_total{status=~"5.."}[1h])
          / rate(http_requests_total[1h])
        ) > (14.4 * 0.001)  # 14.4x burn rate on 1h window
      for: 5m
      labels:
        severity: critical
      annotations:
        summary: "High error rate burning SLO budget"
        runbook: "https://runbooks.example.com/api-errors"

OpenTelemetry — The Unifying Standard

📘 OpenTelemetry

OpenTelemetry (OTel) is the CNCF project that standardizes how telemetry is collected and exported. It provides vendor-neutral SDKs, APIs, and a Collector. Instrument once with OTel and route to any backend — Jaeger today, Honeycomb tomorrow. It has become the de-facto standard in 2024+.

MODULE 08

Lab & Best Practices

⏱ ~5 minutes · Putting It All Together

Observability Maturity Model

Level 1 — Reactive (No observability)

Users report problems. Log in via SSH. Grep logs manually. No alerting.

Level 2 — Basic Monitoring

Uptime checks, CPU/memory dashboards. Basic alerting on thresholds. Centralized logs.

Level 3 — Structured Observability

Three pillars instrumented. Structured logs with trace IDs. Distributed tracing. SLOs defined.

Level 4 — Proactive

Anomaly detection. Error budget burn rate alerts. Correlated signals. On-call runbooks linked to alerts.

Level 5 — Predictive / AIOps

ML-based root cause analysis. Auto-remediation. Observability data drives architecture decisions.

🧪 Hands-On Lab: Instrument a Service

🧪 Lab Exercise

Take the code below and add: (1) a request counter, (2) a latency histogram, (3) structured logging with trace_id, (4) an OTel span. Then query the p99 latency in Prometheus and link a log to its trace in Grafana.

python fastapi service — add observability to this
from fastapi import FastAPI, Request
from prometheus_client import Counter, Histogram, make_asgi_app
from opentelemetry import trace
import structlog, time

app = FastAPI()
tracer = trace.get_tracer(__name__)
logger = structlog.get_logger()

# 📊 METRICS
REQUEST_COUNT = Counter(
    "http_requests_total",
    "Total requests",
    ["method", "endpoint", "status"]
)
REQUEST_LATENCY = Histogram(
    "http_request_duration_seconds",
    "Request latency",
    ["endpoint"],
    buckets=[.005,.01,.025,.05,.1,.25,.5,1,2.5]
)

@app.get("/users/{user_id}")
async def get_user(user_id: str, request: Request):
    start = time.time()
    
    # 🔗 TRACE — create span, get trace_id for logs
    with tracer.start_as_current_span("get_user") as span:
        span_ctx = span.get_span_context()
        trace_id = format(span_ctx.trace_id, "032x")
        span.set_attribute("user.id", user_id)
        
        # 📄 LOG — structured with trace correlation
        logger.info(
            "get_user_request",
            user_id=user_id,
            trace_id=trace_id,
            method=request.method
        )
        
        user = await db.get_user(user_id)
        
        duration = time.time() - start
        
        # Record metrics
        REQUEST_COUNT.labels("GET", "/users", 200).inc()
        REQUEST_LATENCY.labels("/users").observe(duration)
        
        return user

The Observability Checklist

✅
Instrumentation

All services emit metrics, structured logs, and traces. OTel SDK used. Auto-instrumentation enabled for DBs, HTTP clients.

✅
Correlation

trace_id in every log line. Metrics dashboards link to traces. Single pane of glass (Grafana/Datadog).

✅
SLOs Defined

Availability and latency SLOs set for all user-facing services. Error budget tracked. Burn rate alerts in place.

✅
Runbooks

Every alert links to a runbook. On-call knows what to do. Post-mortems feed back into improved alerting.
