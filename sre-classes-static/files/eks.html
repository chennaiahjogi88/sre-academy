AWS

EKS vs Self-Managed Kubernetes

Why Choose Amazon EKS?

Managed control plane (etcd, API server, scheduler)
Native AWS integrations (IAM, VPC, ALB, RDS)
Automatic updates and patches
99.95% SLA availability guarantee
Multi-AZ redundancy
No master node management
You pay for EC2 nodes and EKS control plane per cluster-hour (~$0.10/hour)
Architecture

EKS Architecture

Control Plane (AWS Managed):

  • Kubernetes API Server
  • etcd (data store)
  • Controller Manager & Scheduler
  • Runs in AWS-managed subnets

Data Plane (Your Responsibility):

  • EC2 worker nodes (or Fargate)
  • Custom VPC with public/private subnets
  • Security groups and IAM roles
  • Kubelet, kube-proxy running on nodes
Control Plane communicates with nodes over VPC networking
Compute

Node Groups in EKS

Three Types of Node Groups:

Managed

AWS manages patching, scaling, updates

Best for: Most workloads

Self-Managed

You control AMI, patches, scaling

Best for: Custom requirements

Fargate

Serverless compute, no nodes

Best for: Burstable workloads

Instance Types:

  • On-Demand - Full price, always available
  • Spot - Up to 90% discount, but interruptible
Security

IAM & IRSA

aws-auth ConfigMap: Maps IAM users/roles to K8s groups

apiVersion: v1 kind: ConfigMap metadata: name: aws-auth data: mapRoles: | - rolearn: arn:aws:iam::ACCOUNT:role/NodeInstanceRole username: system:node:{{EC2PrivateDNSName}}

IRSA (IAM Roles for Service Accounts):

  • Pods assume IAM roles via OIDC provider
  • Fine-grained permissions per pod
  • More secure than sharing node credentials
Pod can call AWS APIs with temporary credentials via IRSA

Operations

Creating an EKS Cluster

Option 1: Using eksctl (easiest):

# Create cluster with default settings eksctl create cluster --name my-cluster --region us-east-1 # Create cluster with custom config eksctl create cluster -f cluster.yaml

Option 2: Using AWS CLI:

# Create EKS cluster aws eks create-cluster --name my-cluster \\ --version 1.27 --role-arn arn:aws:iam::ACCOUNT:role/eks-service-role \\ --resources-vpc-config subnetIds=subnet-xxx,subnet-yyy # Update kubeconfig aws eks update-kubeconfig --name my-cluster --region us-east-1 # Verify connection kubectl get nodes
Quiz

Knowledge Check

Q1: What is the main advantage of EKS over self-managed K8s?
AWS manages the control plane (etcd, API server, scheduler), patches, updates, and provides 99.95% SLA.
Q2: What are the three node group types in EKS?
Managed (AWS handles patching/scaling), Self-Managed (you control everything), Fargate (serverless).
Q3: What is IRSA and why use it?
IAM Roles for Service Accounts - allows pods to assume IAM roles via OIDC, giving fine-grained AWS permissions per pod.
Q4: Name 2 essential EKS managed add-ons:
CoreDNS (DNS), kube-proxy (networking), VPC CNI (container networking), EBS CSI Driver (volumes).
Q5: What tool makes creating EKS clusters easiest?
eksctl - a CLI tool by Weaveworks that simplifies EKS cluster creation with YAML configuration.
